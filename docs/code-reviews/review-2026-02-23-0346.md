# Code Review Report: SignalNest (mr_notifier) - 2026-02-23

## Executive Summary
SignalNest is an ambitious event intelligence system designed to process GitHub/Discord events using a custom DSL. Architecturally, the project is well-planned, featuring a sophisticated canonical tree database schema and a modular server-side structure. However, the current implementation is significantly incomplete. The server relies heavily on placeholders for core features, and the Android application contains a critical logic flaw that prevents it from functioning as intended.

**Overall Grade: D (Technical Prototype)**
*Note: While the architectural plan is A-grade, the actual implementation is currently at a pre-alpha/prototype level.*

---

## Critical Issues (Blocking Production)

### 1. Non-Functional Notification Scheduling (Android)
**File:** `mr_notifier/app/src/main/java/fury/notifier/notification/AppNotificationManager.kt`
The app attempts to schedule notifications using `Handler(context.mainLooper).postDelayed` and then immediately terminates the Activity with `finishAndRemoveTask()`. 
- **Impact:** Once the app process is terminated, the Handler callback is cleared, and the notification is never shown. 
- **Requirement:** Must use `AlarmManager` or `WorkManager` for persistent scheduling.

### 2. Missing Internet Permission (Android)
**File:** `mr_notifier/app/src/main/AndroidManifest.xml`
The manifest lacks the `INTERNET` permission.
- **Impact:** The app cannot communicate with the SignalNest server, making the entire sync architecture useless.
- **Requirement:** Add `<uses-permission android:name="android.permission.INTERNET" />`.

### 3. Core Engine Missing (Server)
**Directory:** `mr_notifier-server/src/dsl/`
The directories for the DSL Lexer, Parser, Compiler, and Executor are empty or contain only placeholders.
- **Impact:** The primary value proposition (DSL-driven event processing) is unimplemented.

---

## Major Issues

### 1. Broken Logic in Mock Mode Check
**File:** `mr_notifier-server/src/modules/auth/auth.routes.ts`
The code checks `if (!isMock)` where `isMock` is the function object itself, which is always truthy.
- **Impact:** The `else` block (mock logic) is always executed for certain operations like updating `last_login`, even when running in production mode.
- **Requirement:** Change to `if (!isMock())`.

### 2. Incomplete Database Initialization
**File:** `mr_notifier-server/src/core/db/database.ts`
The `initDatabase()` function logs "Database initialized" but contains no logic to actually execute `schema.sql`.
- **Impact:** Fresh deployments to Supabase will fail as tables won't be created automatically.

### 3. Missing Test Coverage
Despite a `vitest` configuration in `package.json`, there are zero unit or integration tests in the `mr_notifier-server` or `mr_notifier` app.

---

## Minor Issues & Technical Debt

- **Unreliable Sync Logic**: Timestamp-based sync (`gt('created_at', last_sync_timestamp)`) is prone to race conditions and clock drift issues. The sophisticated `sync_version` column in the schema is not yet fully utilized in the logic.
- **Placeholders**: Multiple API routes (GitHub webhooks, notifications list) are just returning "coming soon" messages.
- **Inconsistent Naming**: The server identifies itself as "SignalNest" in logs/package.json, but the app is "Notifier 300".
- **Logging Performance**: The logger uses `JSON.stringify` on every call, which could be a bottleneck if large event payloads are logged frequently.

---

## Long-term Risks

- **Feature Bloat vs. Core Stability**: The database schema is extremely broad (tracking commits, stars, forks, issues, etc.), but without a working sync engine, this creates a massive maintenance surface before the core value is proven.
- **Process Termination**: On modern Android versions, background processes are aggressively killed. Relying on simple scheduling without Foreground Services or robust WorkManager integration will lead to poor reliability.

---

## Suggested Roadmap Updates

1.  **Immediate Fix**: Replace `Handler` with `WorkManager` in the Android app and add `INTERNET` permission.
2.  **Core Development**: Prioritize the DSL Lexer/Parser to enable basic rule processing.
3.  **Reliability**: Implement the `increment_sync_version` triggers properly in the database and switch the sync API to use version numbers instead of timestamps.
4.  **Automation**: Add basic unit tests for the Auth logic and the DSL components once implemented.
