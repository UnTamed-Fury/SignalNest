# Code Audit Report: SignalNest (mr_notifier)

**Date:** 2026-02-23 03:10 UTC
**Auditor:** Gemini CLI (Scheduled 10-minute Static Audit)
**Scope:** Full Repository (Server + Android)

---

## 1. Executive Summary
SignalNest is an "Event Intelligence System" comprising a Hono-based TypeScript server and a modern Jetpack Compose Android application. The project is currently in early MVP stage. While the architectural choices (Hono, Supabase, Jetpack Compose, MVVM) are solid and modern, the implementation is currently fragmented with significant gaps between the stated vision (DSL-driven intelligence) and the actual code (empty DSL folders). Core functionality relies heavily on in-memory storage on both server and client, posing high risks for data persistence.

---

## 2. Critical Issues

### 2.1. Lack of Data Persistence (Android)
The `SignalNestRepository` in the Android app uses `MutableStateFlow` for all primary data (notifications, notes, todos). This data is lost immediately upon application termination. 
*   **Impact:** Zero reliability for a notification/task management system.
*   **Recommendation:** Prioritize the implementation of Room database for local persistence immediately.

### 2.2. In-Memory Mock Store (Server)
The server defaults to an in-memory `Map` when Supabase credentials are not provided. While acceptable for rapid local prototyping, it lacks any persistence between server restarts.
*   **Impact:** Data loss during deployments or crashes.
*   **Recommendation:** Ensure Supabase or a local SQLite/PostgreSQL instance is strictly required for any environment beyond initial local dev.

### 2.3. Zero Test Coverage
No unit, integration, or instrumentation tests were found in either the server or the Android codebase, despite dependencies (vitest, junit) being present.
*   **Impact:** High regression risk; impossible to verify correctness of sync or auth logic automatically.
*   **Recommendation:** Implement core unit tests for the `auth` module and the `SignalNestRepository` sync logic.

---

## 3. Major Issues

### 3.1. Theoretical DSL Engine
The repository contains empty directories for a DSL engine (`lexer`, `parser`, `compiler`, etc.), which is described as a core feature. 
*   **Impact:** The "Event Intelligence" aspect of the project is currently non-existent.
*   **Recommendation:** Define the DSL grammar and implement a basic lexer/parser if this remains a core value proposition.

### 3.2. Repetitive Auth Logic
The `authRoutes.get('/me')` endpoint manually verifies JWT tokens instead of using a reusable Hono middleware.
*   **Impact:** Maintainability overhead and increased risk of security inconsistencies.
*   **Recommendation:** Refactor JWT verification into a middleware.

---

## 4. Minor Issues

### 4.1. Broad CORS Configuration
CORS is currently set to `*`. 
*   **Recommendation:** Restrict CORS to specific origins in production.

### 4.2. Roadmap Inconsistency
The `ROADMAP.md` refers to an Express-based backend, while the implementation has moved to Hono.
*   **Recommendation:** Update documentation to reflect current tech stack (Hono, Supabase).

### 4.3. Placeholder Screens (Android)
Several core UI components (Sources, Schemas, Logs) are currently just `PlaceholderScreen` instances.

---

## 5. Technical Debt
- **Room Migration:** Pending on Android.
- **Supabase Schema:** Currently handled via manual `schema.sql` rather than a migration tool.
- **Auth Middleware:** Missing on server.
- **Error Handling:** Minimal; relies on a global `onError` handler without specialized error types.

---

## 6. Suggested Roadmap Updates
1.  **Immediate:** Implement Room (Android) and ensure Supabase persistence (Server).
2.  **Next:** Add Auth Middleware and initial Unit Tests.
3.  **Short-term:** Bridge the gap between the UI and the (currently empty) DSL module.
4.  **Medium-term:** Replace in-memory logic in `SignalNestRepository` with a single source of truth (Room).

---

## 7. Score: 45/100
*   **Architecture:** 15/20 (Modern stack, clean modules)
*   **Security:** 10/20 (Bcrypt/JWT used, but implementation is repetitive and CORS is too open)
*   **Maintainability:** 10/20 (Clean code, but no tests)
*   **Production Readiness:** 5/20 (Critical persistence issues)
*   **Completeness:** 5/20 (Many placeholders and empty core modules)

**Verdict:** Promising foundation but currently in a "Pre-Alpha" state due to critical persistence and testing gaps.
