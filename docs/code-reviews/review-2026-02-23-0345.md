# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Version:** 3.0.0 (Server) / 0.1.0 (Documentation)

## Executive Summary
The SignalNest project (formerly mr_notifier) is in an early, highly volatile stage of development. While the core vision of an "Event Intelligence System" is ambitious, the current implementation is essentially a skeleton with significant security holes and zero test coverage. The system is split between a Hono-based Node.js server and a Jetpack Compose Android application. The server features a "Mock Mode" for development without a database, but this fallback mechanism is fragile.

## Critical Issues
1. **Unprotected API Endpoints**: The `/api/notifications` and `/api/sync` endpoints lack authentication middleware. Any user can access, create, or sync notifications for any `user_id` by simply providing the ID in the request.
2. **Hardcoded Credentials Leak**: The `mr_notifier-server/.env.template` file contains a REAL Supabase PostgreSQL connection string, including a password. This is a massive security breach.
3. **Stateless Android App**: The Android application (`forked-version`) uses in-memory storage (StateFlow) for all notifications, notes, and todos. All user data is lost when the app is killed or the device restarts.
4. **Zero Test Coverage**: There are no local tests (unit, integration, or E2E) for either the server or the Android application.

## Major Issues
1. **Inconsistent Documentation**: The `README.md` refers to a FastAPI backend, but the actual implementation is Hono (Node.js). Version numbers and feature lists are mismatched across the repository.
2. **Missing DSL Engine**: The "DSL Engine" touted in the project description and `package.json` consists of empty directories. No event processing logic is actually implemented.
3. **Wide-Open CORS**: The server implements `Access-Control-Allow-Origin: *`, which is unsuitable for a production system handling private notification data.
4. **Input Validation Gaps**: While Zod is used for environment variables, it is mostly ignored for route parameters and request bodies, leading to potential type errors and crashes (e.g., `parseInt` on unchecked query params).

## Minor Issues
1. **Naming Inconsistencies**: The codebase mixes `snake_case` (e.g., `user_id`, `last_sync_timestamp`) and `camelCase` (e.g., `userId`, `isRead`) inconsistently between the API and internal logic.
2. **Fragile Mock Mode**: The server automatically falls back to Mock Mode if credentials are missing, which could lead to silent data loss if production environment variables are misconfigured.
3. **Redundant Repositories**: The `mr_notifier/` directory contains an extremely simplified version of the app that conflicts with the more complete `forked-version/`.

## Long-Term Risks
- **Scalability**: The current sync logic (fetching all notifications since a timestamp) will become a performance bottleneck as notification history grows.
- **Maintainability**: Without tests and with such frequent architectural shifts (FastAPI -> Hono), the project is at high risk of regression.
- **Data Integrity**: The lack of a local database in the Android app prevents any offline-first features and makes the app unreliable for a notification system.

## Suggested Roadmap Updates
1. **Security Hardening**:
   - Implement JWT middleware for all `/api/*` routes.
   - Immediately rotate all leaked credentials and remove them from `.env.template`.
   - Restrict CORS origins.
2. **Persistence**:
   - Integrate Room database into the Android app for local persistence.
   - Implement proper database migrations for the server.
3. **Testing**:
   - Set up Vitest for server-side logic and DSL components (once implemented).
   - Add Android unit tests for the Repository and ViewModel logic.
4. **Infrastructure**:
   - Standardize naming conventions across the stack.
   - Align documentation with the current tech stack.

## Score: 35/100
*The project has a clear vision and basic connectivity but fails on fundamental production standards (Security, Persistence, Testing).*
