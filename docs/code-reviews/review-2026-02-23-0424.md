# Code Audit Report: SignalNest (Notifier 300)
**Date:** 2026-02-23 04:30
**Auditor:** Gemini CLI (Autonomous Audit Mode)
**Status:** Pre-Alpha / MVP In-Progress

---

## Executive Summary
The project demonstrates strong architectural planning, particularly in its database schema and roadmap definitions. However, there is a significant discrepancy between the claimed progress in documentation and the actual implementation in the codebase. The server is currently a skeleton with several security vulnerabilities (BOLA, missing auth middleware), and the "core" value proposition (DSL engine, external adapters) consists of empty directories. The Android application is split between a minimal demo and a more complete fork, neither of which is production-ready.

---

## Critical Issues (Must Fix Immediately)
1. **Broken Object Level Authorization (BOLA)**: The server endpoints for notifications and sync (`GET /api/notifications`, `POST /api/sync`, etc.) accept a `user_id` parameter from the client and use it directly to query the database without verifying it against the authenticated user's ID from the JWT token.
2. **Missing Authentication Middleware**: Although JWT logic is implemented in `auth.routes.ts`, it is not applied as global or route-specific middleware in `index.ts`. Consequently, all API routes are currently unprotected.
3. **Discrepancy in Roadmap vs. Code**: The `ROADMAP.md` marks Phase 1 as COMPLETED, including "Notification storage" and "Basic sync". However, the implementation relies on a volatile in-memory "Mock Mode" that loses all data on restart and lacks proper conflict resolution.

---

## Major Issues
1. **Empty Core Modules**: The `dsl/` (compiler, executor, parser) and `adapters/` (github, discord, webhook) directories in the server are empty skeletons. The core logic for event processing does not yet exist.
2. **Unreliable Notification Scheduling (Android)**: The `mr_notifier` app uses `Handler.postDelayed()` for scheduling. This will not survive app death or System Doze mode. `WorkManager` or `AlarmManager` must be used (partially addressed in the `forked-version` but missing in the main app).
3. **Code Duplication & Ambiguity**: The existence of two separate Android applications (`mr_notifier` and `forked-version`) with different package names and implementation levels creates significant maintenance overhead and confusion.
4. **Lack of Automated Tests**: Zero unit, integration, or UI tests were found despite having test frameworks (`vitest`, `junit`) in the configuration files.

---

## Minor Issues
1. **In-memory Storage**: Both the server (in mock mode) and the Android app (in the forked version) rely on in-memory Maps/Flows. This is unsuitable for an "offline-first" application.
2. **Naming Inconsistency**: The database schema uses `snake_case` (e.g., `user_id`), while the mock store and some API models use `camelCase` (e.g., `userId`).
3. **Hardcoded Strings**: Several UI strings and log messages are hardcoded instead of using resource files or structured logging.

---

## Long-term Risks
1. **Scalability of DSL**: The intended DSL engine is complex. Without a robust execution environment (e.g., sandboxed VM), it may introduce security risks or performance bottlenecks.
2. **Maintenance of Adapters**: Supporting multiple external platforms (GitHub, Discord, etc.) will require constant maintenance as external APIs change.
3. **Data Integrity**: The current "Sync" logic is primitive and will likely fail or cause data duplication in multi-device scenarios without a proper versioning/tombstone strategy.

---

## Suggested Roadmap Updates
1. **Consolidate Android Apps**: Immediately merge the `forked-version` logic into the main `mr_notifier` app and delete the redundant code.
2. **Implement Auth Middleware**: Secure all `/api/*` routes before proceeding with feature development.
3. **Database Persistence**: Move from "Mock Mode" to a real PostgreSQL/Supabase instance and implement Room on Android.
4. **Core Logic Implementation**: Prioritize the `webhook` adapter and a basic version of the DSL engine.

---

## Audit Score: 38/100
*   **Architecture:** 80/100 (DB schema is excellent)
*   **Security:** 10/100 (Critical BOLA and missing middleware)
*   **Implementation:** 20/100 (Core logic is missing, empty folders)
*   **Maintainability:** 30/100 (Conflicting app versions, missing tests)
*   **Documentation:** 50/100 (Roadmap exists but is misleading)

**Verdict:** NOT production-ready. Significant engineering effort required to bridge the gap between architectural vision and implementation reality.
