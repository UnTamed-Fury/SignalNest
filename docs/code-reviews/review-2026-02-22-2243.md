# Static Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-22
**Auditor:** Gemini CLI
**Project Version:** 3.0.0 (Server) / Refactored (App)
**Overall Score:** 42/100 (High Potential, Low Maturity)

---

## 1. Executive Summary
SignalNest is an ambitious event intelligence system designed to process external events (GitHub, Discord, Webhooks) via a custom DSL and deliver intelligent notifications to a mobile app. The architectural vision, particularly the database schema and sync logic, is impressive and approaches production standards. However, the current code implementation is in a "prototype" state with critical security vulnerabilities, stability bugs, and significant technical debt (e.g., in-memory storage on mobile).

---

## 2. Critical Issues (Action Required Immediately)

### 2.1 Missing Authentication Middleware on API Routes
The server defines `/api/notifications` and `/api/sync` routes but does not apply any authentication middleware in `index.ts`. Furthermore, these routes accept `user_id` as a query parameter and use it directly without verifying if it matches the authenticated user.
- **Risk:** Any user can read, modify, or delete notifications for any other user.
- **Location:** `mr_notifier-server/src/index.ts`, `mr_notifier-server/src/modules/notifications/notifications.routes.ts`.

### 2.2 Production Crash: Undefined Variables
In `notificationRoutes.get('/')`, the Supabase query references `start` and `end` variables which are block-scoped inside the `isMock()` branch.
- **Risk:** The server will crash with a `ReferenceError` when running in production (Supabase) mode.
- **Location:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts:42`.

### 2.3 Security Misconception: Shared JWT Secret
The Android app's `ApiClient` stores a `JWT_SECRET` in its DataStore. The server uses the same secret to sign tokens.
- **Risk:** If the app stores the server's signing secret, a compromised app or extracted APK allows an attacker to sign their own valid JWTs for the server, bypassing all authentication.
- **Location:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`.

## 3. Major Issues

### 3.1 Authentication Logic Bug
In `auth.routes.ts`, the check `if (!isMock)` is missing parentheses for the function call `isMock()`. Since `isMock` is a function (truthy), `!isMock` is always `false`.
- **Impact:** The `last_login` timestamp is never updated in Supabase mode.
- **Location:** `mr_notifier-server/src/modules/auth/auth.routes.ts:133`.

### 3.2 In-Memory Repository (Mobile)
The Android app currently uses `MutableStateFlow` in a repository for all notifications, notes, and todos.
- **Impact:** All user data is lost when the app process is killed or the phone restarts. This is unacceptable for a production notification/productivity tool.
- **Location:** `forked-version/app/src/main/java/com/signalnest/app/data/repository/SignalNestRepository.kt`.

### 3.3 Inefficient Network Usage
`HomeViewModel.markAllAsRead()` iterates through local notifications and makes individual `markAsRead` network calls instead of using the single batch `mark-all-read` endpoint provided by the server.
- **Impact:** High latency and battery drain for users with many unread notifications.
- **Location:** `forked-version/app/src/main/java/com/signalnest/app/ui/home/HomeViewModel.kt:91`.

## 4. Minor Issues & Technical Debt

### 4.1 "Vaporware" DSL Engine
The project structure contains directories for Lexer, Parser, and Compiler, but they are currently empty. The "DSL Engine" is a core USP of the project but hasn't been implemented yet.
- **Location:** `mr_notifier-server/src/dsl/`.

### 4.2 Stale API Instance (Mobile)
`ApiClient.api` is a `lazy` property. If `baseUrl` is changed in settings, the `api` instance remains bound to the old URL until the app restarts.
- **Location:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`.

### 4.3 Missing CI/CD & Tests
The server has `vitest` in `package.json` but no actual tests were found. There are no GitHub Actions for automated testing or linting, only for building.

---

## 5. Long-term Risks
- **Scalability:** The current sync logic (fetching all notifications) will degrade as user data grows. The `sync_version` logic in the schema is a good start but needs implementation in the API.
- **Maintainability:** The gap between the database schema and the code suggests a "design-first, implement-later" approach that might lead to inconsistencies if not synchronized soon.

---

## 6. Suggested Roadmap Updates
1. **Immediate:** Implement a Hono middleware to verify JWTs on all `/api/*` routes and inject `userId` into the context.
2. **Immediate:** Fix the `start/end` variable crash in notification routes.
3. **Short-term:** Replace in-memory storage in the Android app with a Room database.
4. **Short-term:** Remove `JWT_SECRET` from the mobile app; it should only handle the `access_token`.
5. **Mid-term:** Implement the DSL engine to move beyond hardcoded GitHub placeholders.
6. **Mid-term:** Implement the versioned sync logic using the `sync_version` column already present in the schema.
