# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Score:** 42/100

---

## Executive Summary
SignalNest is an ambitious notification intelligence system consisting of a Hono-based TypeScript server and a Jetpack Compose Android application. While the project demonstrates a sophisticated architectural vision (especially in its database schema), the current implementation is in a **highly volatile, pre-alpha state**. The codebase suffers from critical security vulnerabilities, significant missing components (DSL engine), and a total lack of automated testing. The "production standard" goal is currently far from reach.

---

## Critical Issues

### 1. Zero Authentication Enforcement (Security)
*   **Issue:** While a JWT authentication module exists, it is **not used as middleware** to protect sensitive endpoints.
*   **Risk:** Any person can access `/api/notifications?user_id=1` or `/api/sync` and read/modify data for any user. The system relies entirely on the client providing the correct `user_id`.
*   **Impact:** Complete data breach potential and lack of multi-tenant isolation.

### 2. Missing Core Components (DSL Engine & Adapters)
*   **Issue:** The project's "DSL Engine" (Lexer, Parser, Compiler, Executor) and GitHub/Discord adapters are referenced in documentation and `package.json` but the source directories are **empty**.
*   **Risk:** The primary unique selling point of the project (Event Intelligence) does not exist in the source code.
*   **Impact:** The system is currently just a basic CRUD notification store.

### 3. Total Absence of Automated Tests
*   **Issue:** There are **zero** unit, integration, or UI tests in either the server or the Android projects.
*   **Risk:** High probability of regressions during refactoring, especially given the dual "Mock/Supabase" logic paths.
*   **Impact:** Unverifiable system stability.

---

## Major Issues

### 1. Tight Coupling and Lack of Service Layer
*   **Issue:** The server implementation places business logic, database queries, and mock logic directly inside Hono route handlers.
*   **Impact:** Hard to maintain, difficult to test (if tests were added), and violates separation of concerns.

### 2. Unreliable Notification Scheduling (Android)
*   **Issue:** The `mr_notifier` app uses `Handler.postDelayed` for notifications.
*   **Risk:** If the app process is killed by the OS (common in Android), pending notifications are lost forever.
*   **Correction:** Should use `AlarmManager` or `WorkManager` for guaranteed delivery. (Note: `forked-version` uses FCM, which is better but doesn't solve local scheduling if intended).

### 3. Duplicate/Conflicting Projects
*   **Issue:** The repository contains `mr_notifier` and `forked-version`. `forked-version` appears to be the "real" project, while `mr_notifier` is a skeleton. 
*   **Impact:** Developer confusion and fragmentation. The legacy project should be archived or merged.

### 4. Technical Debt: Mock Logic Pollution
*   **Issue:** Every route is polluted with `if (isMock())` blocks.
*   **Impact:** The codebase is twice as large as it needs to be and prone to bugs where mock behavior differs from production behavior. 
*   **Suggested Fix:** Use the Strategy pattern or Dependency Injection to swap between `MockRepository` and `SupabaseRepository`.

---

## Minor Issues

*   **In-Memory Storage on Android:** Due to Termux limitations, the app uses in-memory storage, meaning all data is lost on app close. This is acceptable for a "Termux-first" dev environment but not for production.
*   **Hardcoded Secrets:** While `env.ts` uses Zod, the `google-services.json` and other configuration files are present in the repo (checked via file tree), which is a security risk if public.
*   **Minimal Documentation:** KDoc is missing for most Android classes, and JSDoc is sparse in the server.

---

## Long-term Risks

1.  **Termux Dependency:** Optimizing the entire architecture for Termux (avoiding Room, kapt, etc.) creates a "technical ceiling" that will make it harder to transition to a standard professional development environment later.
2.  **Schema Over-engineering:** The database schema is extremely complex (15+ tables) compared to the current logic (2 tables). This might lead to maintenance overhead as the logic catches up to the schema.

---

## Suggested Roadmap Updates

1.  **IMMEDIATE:** Implement an `authMiddleware` in the server and apply it to all `/api/*` routes.
2.  **IMMEDIATE:** Set up a basic testing framework (Vitest for server, Espresso/Robolectric for Android) and write at least one smoke test.
3.  **Refactor:** Extract business logic from routes into `Services` and database logic into `Repositories`.
4.  **Implementation:** Actually implement the SNRL Lexer and Parser if they are indeed intended features.
5.  **Reliability:** Replace `Handler.postDelayed` with `WorkManager` in the Android app.
