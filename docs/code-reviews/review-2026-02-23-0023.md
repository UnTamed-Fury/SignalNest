# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Score:** 45/100

---

## 1. Executive Summary
SignalNest is an ambitious "Event Intelligence System" that is currently in a "Design-First, Implementation-Second" phase. The project demonstrates a high level of architectural maturity in its database design and entity modeling, but the actual codebase is largely composed of skeletons, placeholders, and mock implementations. 

A significant constraint is the development environment (Termux), which has led to critical architectural compromises in the Android application, such as the abandonment of persistent local storage (Room). On the server side, the core USP of the project—the DSL Engine—exists only as an empty directory structure.

---

## 2. Critical Issues

### 2.1. Zero Test Coverage
Across the entire repository (Server, Android App, Forked App), there are **zero** automated tests. 
- **Impact:** High. Any change is prone to regressions. The project cannot be considered stable or production-ready without a verification layer.
- **Recommendation:** Implement Vitest for the server and JUnit/Compose UI tests for the Android app. Start with critical paths like Authentication and Sync.

### 2.2. Missing Core Functionality (DSL Engine)
The project description highlights a "DSL Engine," and the directory structure/config for it exists, but all implementation folders (`lexer`, `parser`, `compiler`, etc.) are empty.
- **Impact:** Critical. The primary differentiator of the product is missing.
- **Recommendation:** Prioritize the implementation of the DSL lexer and parser.

### 2.3. Android In-Memory Storage
The Android app (forked-version) uses an in-memory repository for notifications, notes, and todos. The `REFACTORING_PLAN.md` states this is because Room/kapt doesn't work on Termux.
- **Impact:** Critical. All user data is lost when the app process is killed.
- **Recommendation:** If Room is unavailable, use **SQLDelight** (which doesn't rely on kapt/ksp in the same way) or a manual SQLite implementation to ensure data persistence.

---

## 3. Major Issues

### 3.1. Client-Side Security Risk (JWT_SECRET)
The `ApiClient.kt` in the Android app contains logic to store and configure a `JWT_SECRET`. 
- **Risk:** If this is the same secret used by the server to sign tokens, an attacker could decompile the APK and forge valid JWTs for any user.
- **Recommendation:** The client should **never** know the server's JWT secret. Remove all secret-related logic from the Android client.

### 3.2. Server implementation Gaps
The `index.ts` and many module directories contain "placeholder" routes and empty files.
- **Impact:** High. Features like GitHub webhooks, delivery logs, and source integrations are defined in the schema but non-existent in the code.
- **Recommendation:** Systematically implement the modules defined in `src/types/entities.ts`.

### 3.3. Version Mismatch
The Android client claims compatibility with server `v3.1.0`, but the server is currently at `v3.0.0`.
- **Impact:** Medium. Leads to API contract mismatches and integration failures.
- **Recommendation:** Sync versions and implement an API versioning strategy (e.g., `/api/v1/...`).

---

## 4. Minor Issues & Technical Debt

- **Hardcoded IDs:** Multiple instances of `user_id: 0` are hardcoded in the Android repository logic.
- **Logger Performance:** The server logger uses `fs.appendFile` on every log call. This is an I/O bottleneck. Use a buffered stream or a library like `pino`.
- **Bcrypt Rounds:** Using 10 rounds for `bcrypt` is acceptable but on the lower end for 2026 security standards. Consider 12.
- **Mock Logic Pollution:** Production routes are heavily cluttered with `if (isMockMode)` checks.
- **Unreachable Scripts:** `package.json` defines `db:init` and `db:migrate` scripts pointing to non-existent files.

---

## 5. Long-term Risks
- **Scalability:** The in-memory `mockUsers` and `mockData` in the server will fail in any horizontally scaled environment.
- **Platform Lock-in:** The heavy reliance on Supabase-specific logic in `database.ts` makes it difficult to migrate to other providers despite the robust SQL schema.
- **Maintenance Burden:** The "forked-version" and "base-version" of the Android app are diverging, creating a maintenance nightmare.

---

## 6. Suggested Roadmap Updates

1. **Phase 1 (Stabilization):** Implement persistence on Android (SQLite/SQLDelight) and basic Unit Tests for Auth.
2. **Phase 2 (Core):** Implement the DSL Lexer and Parser.
3. **Phase 3 (Integration):** Implement the GitHub Webhook adapter and Raw Event processing.
4. **Phase 4 (Security):** Remove `JWT_SECRET` from client and implement proper Refresh Token rotation.
