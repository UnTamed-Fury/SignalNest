# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23 05:40 UTC
**Auditor:** Gemini CLI (Autonomous Audit)

## 1. Executive Summary
The SignalNest project (formerly `mr_notifier`) is currently in a transitional state, likely undergoing a major refactoring to "industry standards" as evidenced by the `forked-version` directory and `REFACTORING_PLAN.md`. While the documentation and server structure (using Hono/Zod) show promise, the codebase suffers from significant gaps: the core DSL engine is missing (empty directories), there are zero automated tests, and several backend routes have critical logic errors and security vulnerabilities. The project is currently at a **Pre-Alpha** stage and does not meet production standards.

---

## 2. Critical Issues (P0)

### 2.1. Missing DSL Engine Core
The `mr_notifier-server/src/dsl` directory contains empty subdirectories (`lexer`, `parser`, `compiler`, etc.). Since the project is marketed as an "Event Intelligence System with DSL Engine," the absence of this core logic is a critical blocker.

### 2.2. Zero Test Coverage
There are **no project-specific tests** across the server, the primary Android app, or the forked version. For a system processing events and notifications, the lack of unit and integration tests makes the codebase extremely fragile and unmaintainable.

### 2.3. Broken Pagination Logic (Server)
In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the `GET /` route in Supabase mode attempts to use `start` and `end` variables that are never defined. This will cause a runtime crash in production environments.

```typescript
// From notifications.routes.ts
.range(start, end); // start and end are undefined
```

---

## 3. Major Issues (P1)

### 3.1. Unprotected API Routes
While authentication routes exist, the notification and sync routes do not currently apply an authentication middleware. Anyone can access, create, or modify notifications if they know the endpoint.

### 3.2. Blind Trust of `userId`
The server blindly trusts `userId` provided in query parameters or request bodies (e.g., `GET /api/notifications?user_id=123`). There is no validation that the requesting user has the authority to view or modify data for that ID.

### 3.3. Dual Android Codebases
The repository contains two separate Android projects (`mr_notifier` and `forked-version`). The `forked-version` appears to be the primary development branch, while the root `mr_notifier` is a functional but extremely basic stub. This creates confusion and maintenance overhead.

---

## 4. Minor Issues (P2)

### 4.1. Unreliable Notification Scheduling
The stub Android app uses `Handler.postDelayed` for notification scheduling. This is an anti-pattern in Android as it will not survive application kills, background restrictions, or device reboots. `WorkManager` or `AlarmManager` should be used.

### 4.2. "Coming Soon" Placeholders
Multiple key screens in the `forked-version` app (Sources, Schemas, Logs) are currently static placeholders. The "Sources" integration (GitHub, Discord) mentioned in the docs is not yet implemented in the code.

### 4.3. Mock Store Inconsistency
The server relies heavily on a `mock-store.ts` for development. While useful, the lack of persistent local storage (e.g., SQLite file) means all data is lost on every server restart, hindering local mobile development.

---

## 5. Long-term Risks
- **Scalability:** The current reliance on in-memory mock stores for development hides potential performance issues with the database schema.
- **Maintainability:** Without tests, refactoring the DSL engine (once implemented) will be dangerous.
- **Security:** The project lacks a robust authorization layer (RBAC), which will be difficult to retrofit later.

---

## 6. Suggested Roadmap Updates
1. **Implement DSL Engine:** Prioritize the lexer and parser to make the system functional.
2. **Infrastructure for Testing:** Set up Vitest for the server and JUnit/Compose Test for Android. Aim for 50% coverage immediately.
3. **Security Lockdown:** Implement and apply a JWT authentication middleware to all `/api/*` routes.
4. **Fix Production Crashes:** Define pagination bounds and resolve the Supabase mode runtime errors.
5. **Consolidate Apps:** Move the `forked-version` to the root and archive the stub `mr_notifier` to avoid confusion.

---

## 7. Score (0–100)
**Score: 35/100**

*Reasoning: While the architectural vision and documentation are strong (A-grade), the actual implementation is missing its core engine, lacks any automated verification, and contains critical runtime errors and security gaps.*
