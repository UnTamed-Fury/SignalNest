# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Version:** 3.0.0 (Server) / 1.0.0 (App)

---

## 1. Executive Summary
The SignalNest project is a notification intelligence system comprising a Node.js/Hono server and a Jetpack Compose Android application. While the architectural vision is ambitious—featuring a custom DSL for event processing and a versioned sync system—the current implementation is in a prototype state with significant gaps. Core features like the DSL engine are entirely missing, and several critical security risks and logic flaws were identified. The project is currently below 'A-grade production standard'.

**Score: 45/100**

---

## 2. Critical Issues (P0)

### 2.1. Security: JWT Secret Exposure
The Android application (`ApiClient.kt` and `SettingsScreen.kt`) allows users to input and store a `JWT_SECRET`. This suggests a fundamental misunderstanding of JWT security. The secret used for signing/verifying tokens must remain exclusively on the server. Client-side knowledge of this secret allows for token forgery and complete system compromise.

### 2.2. Security: Authentication Logic Bug
In `mr_notifier-server/src/modules/auth/auth.routes.ts`, the code checks `if (!isMock)` instead of calling the function `isMock()`. Since `isMock` is a function, `!isMock` is always `false`, causing the "Real Supabase" code path to be skipped during login even when configured.

### 2.3. Logic: Undefined Variables in Server Pagination
In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the Supabase query uses `.range(start, end)`, but `start` and `end` are never defined. This will cause the list notifications endpoint to crash in production (Supabase mode).

### 2.4. Missing Feature: DSL Engine Implementation
The directories for the DSL engine (`lexer`, `parser`, `compiler`, etc.) are present but empty. The core value proposition of the project (event intelligence via DSL) is currently non-existent in the source code.

---

## 3. Major Issues (P1)

### 3.1. Persistence: Missing Android Database
The Android repository (`SignalNestRepository.kt`) uses in-memory storage (Kotlin `StateFlow` and `Map`). All notifications, notes, and todos are lost when the app process is killed. The implementation of Room or a similar persistence library is a critical requirement.

### 3.2. Architecture: Hardcoded User Identity
The Android application hardcodes `userId = 0` for all API calls. While this may be a placeholder for a "no-login" mode, it creates a rigid architecture that conflicts with the multi-user support implemented in the server's authentication and database schema.

### 3.3. Architecture: Android Permission Gaps
The `mr_notifier` version of the app manifest is missing the `INTERNET` permission, which is required for its primary function of syncing with a server. (Note: The `forked-version` includes it, indicating inconsistent development paths).

---

## 4. Minor Issues (P2)

### 4.1. Documentation: Version Mismatch
The Android API service (`SignalNestApi.kt`) claims to be updated for server v3.1.0, while the server's `index.ts` reports version 3.0.0.

### 4.2. Database: Unused SQL Functions
The `increment_sync_version()` PL/pgSQL function in `schema.sql` is defined but not attached to any triggers, meaning the "Versioned Sync" feature is currently manual or non-functional.

### 4.3. Performance: In-memory Mock Store
The server's mock store uses simple in-memory Maps without any cleanup or size limits, which could lead to memory exhaustion during prolonged local testing.

---

## 5. Long-term Risks
- **Scalability**: The current sync logic (`sync_version`) is partially implemented. Without a robust event-sourcing or transaction-log approach, maintaining consistency between multiple mobile clients and the server will become difficult.
- **Maintainability**: The existence of two separate Android app directories (`mr_notifier` and `forked-version`) with diverging logic (e.g., permissions, service definitions) creates a significant maintenance burden.

---

## 6. Suggested Roadmap Updates
1. **Fix Critical Security Flaws**: Remove JWT secret from the client and fix the `isMock()` bug.
2. **Implement Persistence**: Migrate the Android repository to use Room DB.
3. **Core Feature Development**: Implement the DSL engine (Lexer/Parser/Executor) to fulfill the project's primary objective.
4. **Unify Android Apps**: Merge `mr_notifier` and `forked-version` into a single, well-maintained codebase.
5. **Fix Server Logic**: Define pagination variables and attach sync version triggers in the database.

---

## 7. Technical Debt Flag
- **Mock Mode Dependence**: Over-reliance on mock mode has masked several production-only bugs (like the undefined `start`/`end` variables).
- **Missing Tests**: No unit or integration tests were found for the DSL (as it's missing) or the core business logic.
