# Code Review: SignalNest (mr_notifier)
**Date:** 2026-02-23 03:34 UTC
**Status:** Scheduled 10-minute Static Audit
**Score: 35/100**

## Executive Summary
SignalNest is a personal event aggregation and notification system. While the project shows promise in its Android UI design and modular server architecture, it is currently in a "hollow" state. Several core modules (Delivery, DSL, Device Registration) are documented as functional in the roadmap but are actually empty or unimplemented in the source code. Most critically, the system contains a catastrophic security flaw where the server's JWT secret is intended to be shared with the Android client.

---

## Critical Issues

### 1. JWT Secret Exposure (Security)
- **File:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`, `SettingsScreen.kt`
- **Description:** The Android app stores and manages the server's `JWT_SECRET` in local preferences. The UI even includes a field for the user to enter this secret.
- **Risk:** Catastrophic. The client should never know the server's signing secret. If the secret is compromised or reversed from the app, an attacker can forge valid JWTs for any user, bypassing all authentication.
- **Fix:** Remove all `JWT_SECRET` references from the client. The server should sign tokens; the client should only store and send the opaque `access_token`.

### 2. Missing Core Delivery Logic (Logic Flaw)
- **File:** `mr_notifier-server/src/modules/delivery/`
- **Description:** The directory is empty. There is no implementation of Firebase Cloud Messaging (FCM) on the server.
- **Risk:** High. The primary value proposition (push notifications) is not actually implemented. The server stores notifications but never "pushes" them to devices.
- **Fix:** Implement the delivery module using `firebase-admin` SDK.

### 3. Missing Device Registration (Architectural)
- **File:** `mr_notifier-server/src/index.ts`, `SignalNestMessagingService.kt`
- **Description:** The `api/devices/register` endpoint is documented but not implemented in the server. The Android app has a `TODO` in `onNewToken`.
- **Risk:** High. Without device registration, the server cannot map FCM tokens to specific users, making targeted notification delivery impossible.
- **Fix:** Implement the device registration module and ensure the Android app sends its token upon startup/refresh.

---

## Major Issues

### 1. Unimplemented "Notification Intelligence" (Technical Debt)
- **File:** `mr_notifier-server/src/dsl/`
- **Description:** The DSL compiler, executor, and parser directories are completely empty. The "Intelligence" advertised in the README is non-existent.
- **Risk:** The project lacks its defining feature of categorized/normalized events.

### 2. Wide-Open CORS (Security)
- **File:** `mr_notifier-server/src/index.ts`
- **Description:** `Access-Control-Allow-Origin` is set to `*`.
- **Risk:** Insecure for production. It allows any website to make requests to the API.

### 3. Total Absence of Tests (Maintainability)
- **Description:** There are zero unit, integration, or UI tests in either the server or the Android repository.
- **Risk:** High regression risk and low long-term viability.

---

## Minor Issues
- **Insecure Default URL:** `ApiClient.kt` defaults to `http://10.0.2.2:3000/`. Production should use `https`.
- **Hardcoded IP:** The emulator IP `10.0.2.2` is hardcoded in multiple places instead of being a build-time configuration.
- **Mock Mode Limitations:** The server's mock mode uses in-memory maps which are not persistent across Vercel serverless executions.

---

## Long-term Risks
- **Scalability:** The current sync logic returns the entire notification history (or filtered by timestamp). This will become a performance bottleneck as the database grows.
- **Reliability:** Without an ACK-retry loop in the delivery module (which is currently empty), notifications will be lost if the device is offline during the initial push.

---

## Suggested Roadmap Updates
1.  **Sprints 1-2 (Security & Core):**
    - Remove JWT Secret from client.
    - Implement `api/devices/register`.
    - Implement FCM delivery on server.
2.  **Sprints 3-4 (Reliability):**
    - Implement SQLite storage on Android (roadmap mentions Room but it's not implemented).
    - Add unit tests for Auth and Notification logic.
3.  **Sprints 5+ (Intelligence):**
    - Implement the DSL for event processing.

---

## Score: 35/100
*The project has a solid UI foundation but is currently non-functional as a notification system and contains severe security anti-patterns.*
