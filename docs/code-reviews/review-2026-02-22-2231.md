# Code Audit Report: SignalNest (mr_notifier)

**Date:** 2026-02-22  
**Auditor:** Gemini CLI (Autonomous Audit)  
**Project:** SignalNest - Event Intelligence & Notification System  
**Overall Score:** 48/100 (Needs Improvement)

---

## 1. Executive Summary
The SignalNest project is in an early MVP phase with a clear architectural vision but significant implementation gaps and critical logic flaws. While the repository structure is well-organized and follows modern modular patterns (Hono/TypeScript on backend, Kotlin/Compose on Android), the core value proposition (DSL engine) is currently non-existent (empty directories). The codebase lacks any form of automated testing, and several "Completed" features contain runtime errors in production paths.

---

## 2. Critical Issues (Must Fix)

### 2.1 Logic Flaws in Production Paths
- **Undefined Variables:** In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the `start` and `end` variables are defined within a `mock` block but referenced in the `Supabase` block. This will cause a `ReferenceError` in production.
- **Incorrect Function Usage:** In `mr_notifier-server/src/modules/auth/auth.routes.ts`, the code checks `if (!isMock)` instead of `if (!isMock())`. Since `isMock` is a function, this check always fails, potentially leading to incorrect branching.
- **State Inconsistency:** The `sync.routes.ts` uses `mockSyncState` (in-memory) to calculate `currentVersion` even in Supabase mode. A server restart will reset sync versions, breaking client-side sync logic.

### 2.2 Complete Absence of Tests
- There are **zero** project-level tests. No unit tests for core logic (Auth, Sync, Notifications) and no instrumented tests for the Android app. This makes the project highly fragile and prevents "A-grade production standard" certification.

---

## 3. Major Issues

### 3.1 "Ghost" Features (DSL Engine)
- The project structure includes `dsl/compiler`, `dsl/lexer`, `dsl/parser`, etc., and the `package.json` claims to be an "Event Intelligence System with DSL Engine". However, these directories are entirely empty. Advertising non-existent core features as "Phase 1 Complete" is a major risk.

### 3.2 In-Memory Dependency
- The backend relies heavily on `mock-store.ts` and in-memory Maps. While a Supabase client is present, the fallback logic is intrusive and clutters the business logic with branching code.

### 3.3 Backend Storage Gap
- The `forked-version` REFACTORING_PLAN acknowledges that backend storage is "Broken" as in-memory data is lost. This contradicts the "Phase 1 Completed" status in the Roadmap.

---

## 4. Minor Issues & Technical Debt

### 4.1 Development Constraints
- The project is being developed on Termux, which has led to the removal of `Room` and `kapt`. This limits the ability to use industry-standard Android persistence and dependency injection (Hilt).
- **Suggestion:** Explore `KSP` as a replacement for `kapt` if supported, or use a thin SQLite wrapper if Room is unavailable.

### 4.2 Documentation
- API documentation exists as placeholders in `index.ts` logs but lacks a formal specification (Swagger/OpenAPI).
- `docs/api/API.md` was not fully evaluated but initial scans show minimal detail.

---

## 5. Long-term Risks
- **Scalability:** The current sync logic (sending all notifications if no timestamp) will not scale as the database grows.
- **Maintenance:** The "Forked vs Main" confusion (two Android apps in one repo) creates overhead. `forked-version` should be merged or promoted to main once stable.
- **Security:** While JWT/bcrypt are used, the lack of rate limiting and proper input validation (Zod is in dependencies but not widely used in routes) leaves the API vulnerable to brute force and injection.

---

## 6. Suggested Roadmap Updates
1. **Immediate:** Fix `ReferenceError` and `isMock()` function calls in server routes.
2. **Short-term:** Implement a minimum 20% test coverage for Auth and Sync logic.
3. **Short-term:** Connect `sync_meta` to the database properly to ensure persistent versioning.
4. **Mid-term:** Move DSL skeleton to a separate branch or implement a basic lexer to validate the concept.
5. **Mid-term:** Stabilize the Android refactor (`forked-version`) and resolve the Termux/Room conflict.
