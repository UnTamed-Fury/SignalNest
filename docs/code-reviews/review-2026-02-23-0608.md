# Code Audit Report: SignalNest (mr-notifier)

**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Status:** ⚠️ CRITICAL ISSUES FOUND
**Score:** 42/100

---

## 1. Executive Summary
SignalNest is an ambitious notification and event intelligence system. While the architectural vision (Canonical tree, DSL engine, versioned sync) is sound and reflected in the database schema, the current implementation is a high-risk prototype. The codebase suffers from critical logic flaws that will cause production crashes, major security vulnerabilities that expose all user data, and significant feature gaps (DSL engine, webhooks). The project is currently in a 'Pre-Alpha' state and does not meet production standards.

---

## 2. Critical Issues (P0)

### 2.1 Production Crash: Undefined Variables in Notification List
In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the `GET /` endpoint for Supabase mode attempts to use `start` and `end` variables which are only defined within the `isMock()` conditional block.
- **Impact:** The server will crash with a `ReferenceError` whenever this endpoint is called in production.
- **Location:** `notificationRoutes.get('/', ...)` line ~37.

### 2.2 Major Security Risk: Unauthenticated Data Access
The `/api/notifications` and `/api/sync` endpoints do not use authentication middleware. They rely on a client-provided `user_id` parameter.
- **Impact:** Any user (or attacker) can access, modify, or delete notifications for ANY other user by simply changing the `user_id` or `id` parameters in API calls.
- **Location:** `sync.routes.ts`, `notifications.routes.ts`.

### 2.3 Feature Disconnect: Missing 'ack' Endpoint
The mobile app (`forked-version`) expects a `POST /api/notifications/{id}/ack` endpoint, but it is not implemented in the server.
- **Impact:** Core app functionality (acknowledging notifications) will fail.

---

## 3. Major Issues (P1)

### 3.1 Zero Test Coverage
There are NO project-specific tests in the backend or mobile applications. The `test` script exists but only runs tests for the `node_modules` (Zod).
- **Impact:** Regression risk is extremely high. Architectural integrity cannot be verified.

### 3.2 Incomplete Implementation (Vaporware)
- **DSL Engine:** The `dsl/` directory contains only empty folders. The core value proposition of the system is missing.
- **GitHub Webhooks:** Only a placeholder endpoint exists.
- **Mock Mode Leakage:** `mockUsers` in `auth.routes.ts` is local to the module and not shared via `mock-store.ts`, leading to data inconsistency in mock mode.

### 3.3 Security: Missing Rate Limiting
Environment variables for rate limiting are defined (`RATE_LIMIT_MAX_REQUESTS`), but no middleware is actually applied in `index.ts`.
- **Impact:** The server is vulnerable to Brute Force (on `/auth/login`) and DoS attacks.

---

## 4. Minor Issues (P2)

### 4.1 Technical Debt: In-memory Storage in Mobile App
The `forked-version` app uses in-memory `StateFlow` lists for notifications, notes, and todos.
- **Impact:** All data is lost when the app process is killed. The "will be replaced by Room" TODO is a significant pending task.

### 4.2 Sync Logic Limitations
The mobile sync logic implements a "Server Wins" strategy that does not handle offline modifications or conflict resolution. Local changes made while offline (like marking as read) will be overwritten by server state during the next sync.

---

## 5. Architectural Weaknesses

1.  **Mock Mode Dependency:** The system relies heavily on a "Mock Mode" that is triggered if environment variables are missing. This is useful for dev but dangerous if a production config error silently drops the system into an in-memory mock state.
2.  **Stateless vs. Stateful:** The backend schema is highly stateful and complex, but the current Hono implementation is almost entirely stateless/passthrough, failing to leverage the power of the designed schema (e.g., event processing, canonical mapping).

---

## 6. Long-term Risks

1.  **Scalability:** The use of `user_id=0` as a catch-all for "no login" will make future multi-tenant scaling difficult.
2.  **Maintainability:** The fragmentation between `mr_notifier` (basic) and `forked-version` (advanced) mobile apps creates confusion and maintenance overhead.
3.  **Security Debt:** Launching with the current unauthenticated sync model will make it very hard to "bolt on" security later without breaking client compatibility.

---

## 7. Suggested Roadmap Updates

1.  **IMMEDIATE:** Fix the `start`/`end` bug in `notifications.routes.ts`.
2.  **IMMEDIATE:** Implement JWT middleware and apply it to ALL `/api/*` routes. Validate `user_id` against the JWT `userId`.
3.  **HIGH:** Implement the DSL Parser and Executor. This is the heart of the project.
4.  **HIGH:** Replace in-memory storage in the mobile app with Room.
5.  **MEDIUM:** Implement proper conflict-resolution sync logic (using the `sync_version` field already in the schema).
6.  **MEDIUM:** Add Vitest unit tests for the DSL logic and Hono integration tests for the API.

---

## 8. Final Score: 42/100
*Grade: F (Not Production Ready)*

The project has a great schema design and a good choice of stack (Hono, Supabase, Kotlin/Compose), but the implementation quality is currently insufficient for any real-world usage.
