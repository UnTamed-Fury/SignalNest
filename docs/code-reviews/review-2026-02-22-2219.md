# Static Code Audit Report: SignalNest (mr_notifier)

**Date:** 2026-02-22  
**Audit Duration:** 10 minutes (Scheduled)  
**Scope:** Full Repository (Server & Android)  
**Auditor:** Gemini CLI (Autonomous Agent)

---

## 1. Executive Summary
The SignalNest project (formerly `mr_notifier`) is currently undergoing a significant architectural refactor. While the **database design and architectural vision are A-grade**, the current **implementation is in a C-grade state**. There is a massive disconnect between the sophisticated PostgreSQL schema and the actual application code, which is riddled with placeholders, block-scoped variable bugs, and critical security risks. The project is heavily constrained by its development environment (Termux), leading to significant technical debt in the Android application.

---

## 2. Critical Issues

### 2.1. Logic Bug: Block-Scoped Variable Crash
In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the `GET /` route for notifications will crash when not in mock mode.
- **Location:** Line 37-41 (Supabase branch)
- **Problem:** The variables `start` and `end` are defined only within the `if (isMock())` block. When `isMock()` is false, the code attempts to use these variables in the `.range(start, end)` call, resulting in a `ReferenceError`.
- **Impact:** Production notification listing is completely broken.

### 2.2. Security Risk: Client-Side JWT Secret
The Android application (`ApiClient.kt`) includes functionality to save and manage a `JWT_SECRET` locally.
- **Problem:** The client should **never** possess the server's JWT secret. This allows a compromised or malicious client to forge authentication tokens for any user.
- **Impact:** Complete compromise of the authentication system.

### 2.3. Security Risk: Permissive CORS
The server (`index.ts`) uses `Access-Control-Allow-Origin: '*'.`
- **Problem:** This allows any website to make requests to the API, increasing the risk of CSRF and unauthorized data access in a production environment.

---

## 3. Major Issues

### 3.1. Zero Test Coverage
- **Status:** No automated tests found in either the server or the Android repository.
- **Impact:** High risk of regressions during the ongoing refactor. The "Testing Strategy" in the REFACTORING_PLAN is currently purely aspirational.

### 3.2. Statelessness & In-Memory Storage
- **Problem:** Both Auth and Notification modules rely heavily on in-memory Maps when in "mock mode".
- **Impact:** Data is lost on every server restart. This hinders realistic development and demoing.

### 3.3. Placeholder Architecture (Vaporware Features)
- **Problem:** The DSL engine (`src/dsl/`) consists of empty directories. GitHub webhooks and Sync endpoints are placeholders.
- **Impact:** The core value proposition of "Event Intelligence" is not yet implemented.

---

## 4. Minor Issues & Technical Debt

### 4.1. Termux Constraints
- **Issue:** Room and Kapt are disabled in the Android app due to Termux compatibility issues.
- **Impact:** The app uses in-memory storage or basic DataStore instead of a robust local database, limiting its offline capabilities and data integrity.

### 4.2. Code Duplication
- **Issue:** The `isMock()` check is manually repeated in almost every route handler.
- **Impact:** Poor maintainability. This logic should be abstracted into a Repository or Service layer.

### 4.3. Project Schizophrenia
- **Issue:** The repository contains two versions (`mr_notifier` and `forked-version`).
- **Impact:** Confusion for developers and potential for deploying the wrong codebase.

---

## 5. Long-term Risks
- **Scalability:** The current mixed logic (Mock vs DB) will become unmanageable as more entities are added.
- **Viability:** Without the DSL engine, the project is just a simple notification forwarder, of which many better-integrated alternatives exist.
- **Security:** The practice of storing secrets in the app and permissive CORS suggests a lack of production-hardening focus.

---

## 6. Suggested Roadmap Updates
1. **Immediate:** Fix the `start`/`end` scoping bug in `notifications.routes.ts`.
2. **Immediate:** Remove `JWT_SECRET` handling from the Android client; use only `AUTH_TOKEN`.
3. **Short-term:** Implement a real local database for Android (search for Room alternatives that work on Termux, like SQLDelight or raw SQLite).
4. **Short-term:** Create a `BaseRepository` to abstract the `isMock()` vs `Supabase` logic.
5. **Mid-term:** Implement the Lexer and Parser for the DSL engine to fulfill the project's core goal.

---
