# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23 03:15 UTC
**Audit Duration:** 10 Minutes
**Scope:** Recursive analysis of `mr_notifier-server` and `forked-version` (Android)

---

## 1. Executive Summary
The SignalNest project is currently in a "late prototyping" phase. While it demonstrates a functional end-to-end flow for notification delivery and synchronization with a modern tech stack (Hono/TypeScript/Supabase + Jetpack Compose), it fails several critical "A-grade production" criteria. Specifically, the project suffers from **major security vulnerabilities**, **zero test coverage**, and **significant missing core features** (DSL Engine). The decision to remove login for immediate usability has introduced a critical privacy risk.

---

## 2. Critical Issues (Blocking Production)

### 2.1 Lack of API Authentication
**File:** `mr_notifier-server/src/modules/sync/sync.routes.ts`, `mr_notifier-server/src/modules/notifications/notifications.routes.ts`
- **Issue:** The `sync` and `notification` endpoints do not validate JWT tokens. Anyone can access or modify any user's notifications by simply providing a `user_id` in the request body or query.
- **Impact:** Total loss of user privacy and data integrity.

### 2.2 JWT Secret Stored in Android Client
**File:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`
- **Issue:** The `JWT_SECRET` is stored in the app's `DataStore`.
- **Impact:** A JWT secret must remain server-side only. If a client has the secret, they can forge valid tokens for any user, rendering the entire authentication system useless.

### 2.3 Zero Test Coverage
- **Issue:** There are no unit, integration, or UI tests in the entire repository source code.
- **Impact:** Extremely high regression risk. The "DSL Engine" and "Sync Logic" are complex and cannot be safely modified without automated verification.

### 2.4 Empty DSL Engine Implementation
**Directory:** `mr_notifier-server/src/dsl/`
- **Issue:** The project markets a "DSL Engine" as a key feature, but the implementation directories are empty.
- **Impact:** Key value proposition is missing.

---

## 3. Major Issues

### 3.1 Architectural Logic Duplication (Mock vs. Real)
**File:** Most server routes.
- **Issue:** Routes contain explicit `if (isMock())` blocks.
- **Impact:** Hard to maintain. Should use a Repository Pattern or Dependency Injection to swap data sources transparently.

### 3.2 In-Memory Only Storage (App & Server Mock)
- **App:** `forked-version` uses in-memory storage due to `kapt` issues on Termux.
- **Server:** "Mock mode" is in-memory only.
- **Impact:** Total data loss on process restart. Unacceptable for a "notification intelligence system" that should be reliable.

### 3.3 Permissive CORS Policy
**File:** `mr_notifier-server/src/index.ts`
- **Issue:** `Access-Control-Allow-Origin` is set to `*`.
- **Impact:** Security risk in production environments.

---

## 4. Minor Issues & Technical Debt

### 4.1 ApiClient.kt Lazy Bug
- **Issue:** The `api` property is initialized lazily with the initial `baseUrl`. If the user changes the URL in settings, the `api` instance remains tied to the old URL.

### 4.2 Versioning Mismatch
- **Issue:** `README.md` (0.1.0) vs `package.json` (3.0.0) vs `do-this.md` (3.1.0).

---

## 5. Long-term Risks
- **Scalability:** The current sync logic (upsert all) will become slow as notification counts grow. Needs a more robust pagination/delta-sync strategy.
- **Maintainability:** Without tests and with logic duplicated across mock/real modes, the codebase will become increasingly fragile.

---

## 6. Suggested Roadmap Updates
1. **Immediate:** Implement JWT middleware on all `/api/*` routes.
2. **Immediate:** Remove `JWT_SECRET` from the Android app.
3. **Short-term:** Implement a Repository Pattern on the server to clean up `isMock()` logic.
4. **Short-term:** Start a test suite (Vitest for server, JUnit/Espresso for App).
5. **Mid-term:** Resolve `kapt` issues or move to a non-Termux build environment to re-enable Room database.
