# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23 02:15:44 UTC
**Auditor:** Gemini CLI
**Scope:** Recursive Static Analysis (Server & Mobile)

---

## Executive Summary
This audit confirms that the SignalNest project remains in a pre-alpha/prototype state despite its versioning (3.0.0). While the project structure is well-organized and follows modern development patterns (Hono/Supabase on server, Jetpack Compose/MVVM on mobile), the **core functional engines (DSL) are entirely missing**, and the **security architecture is non-existent** for critical data endpoints. Significant logic bugs in the synchronization engine further degrade the project's viability.

---

## Critical Issues 🔴

### 1. Missing Authentication Middleware (Unauthorized Data Access)
The server defines `/api/notifications` and `/api/sync` routes in `index.ts` but does not apply any authentication middleware.
- **Impact:** Any user or bot can query, create, or mark notifications as read for *any* `user_id` by simply passing it as a query parameter or in the JSON body.
- **Risk:** Extreme. Full exposure of user notification data.
- **Action:** Implement a JWT verification middleware and apply it to all `/api/*` routes. Ensure `user_id` is extracted from the JWT, not trusted from request parameters.

### 2. Logic Bug: Sync Version Corruption
In `mr_notifier-server/src/modules/sync/sync.routes.ts`, the "Real Supabase mode" incorrectly references `mockSyncState` (an in-memory Map) to calculate the `currentVersion`.
```typescript
const currentVersion = (mockSyncState.get(user_id)?.currentVersion || 0) + 1;
await supabase.from('sync_meta').upsert({ user_id, current_version: currentVersion, ... });
```
- **Impact:** When the server restarts, `mockSyncState` is cleared. The next sync will attempt to upsert `version 1` into Supabase, potentially causing conflicts or breaking client-side incremental sync logic.
- **Risk:** High. Data integrity and sync consistency are compromised.

### 3. Total Absence of Persistence (Android)
As identified in previous audits, the `SignalNestRepository` remains entirely in-memory.
- **Impact:** App is unusable as a notification history tool. All data is purged on process death.
- **Risk:** High. This is a blocker for "A-grade production standard."

### 4. Zero Test Coverage
The repository contains **no functional tests** (0 files found matching `**/*.{test,spec}.{ts,kt}`).
- **Impact:** Extremely high maintenance cost and high probability of regressions.
- **Risk:** High.

---

## Major Issues 🟡

### 1. Empty DSL Engine
The `src/dsl/` directory on the server contains 5 subdirectories, all of which are empty. The project's primary value proposition (Event Intelligence via DSL) is currently vaporware.
- **Risk:** Business/Project Viability.

### 2. Inefficient Batch Operations (Mobile)
The `HomeViewModel.markAllAsRead()` iterates through all local notifications and launches a separate coroutine/network call for each unread notification.
- **Impact:** Significant battery drain, network congestion, and potential rate-limiting triggers on the server.
- **Risk:** Performance/Scalability.

### 3. Duplicated Logic & Violation of DRY
The server routes contain massive `if (isMock()) { ... } else { ... }` blocks.
- **Impact:** High maintenance burden and inconsistent behavior between mock and production environments.
- **Action:** Abstract database operations into a Repository or Service layer.

---

## Minor Issues 🟢

### 1. Inconsistent "Placeholder" Documentation
`index.ts` labels endpoints as "coming soon" or "placeholder," while some are actually implemented in their respective module files. This leads to developer confusion.

### 2. Missing FCM Token Registration
The `onNewToken` method in `SignalNestMessagingService` is a TODO. Push notifications cannot be targeted to devices.

---

## Long-term Risks
- **Architectural Debt:** The lack of a service layer in the backend will make implementing complex DSL-driven notification routing extremely difficult.
- **Development Environment Constraints:** Optimizing for Android/Termux development (avoiding Room/kapt) is severely limiting the quality of the production Android application.

---

## Suggested Roadmap Updates
1. **Security First:** Add JWT middleware to the Hono server immediately.
2. **Sync Fix:** Correct the `currentVersion` logic to query Supabase for the last version before incrementing.
3. **Persistence:** Implement a persistence layer in the Android app. If Room is problematic on Termux, use a simpler KV store (like `DataStore` for basic state) or a manually managed SQLite database.
4. **DSL Implementation:** Begin implementing the Lexer and Parser to move the project beyond a simple CRUD notification app.
5. **Testing:** Establish a "one-test-per-feature" rule to begin building coverage.
