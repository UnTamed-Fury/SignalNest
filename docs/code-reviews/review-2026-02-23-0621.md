# Code Audit Report - SignalNest (mr_notifier)
**Date:** 2026-02-23 06:21:52 UTC
**Audit Type:** Scheduled 10-minute Static Code Audit
**Status:** In-Refactoring (Transitioning from `forked-version` to `mr_notifier`)

## Executive Summary
SignalNest is a notification management and event intelligence system consisting of a Hono-based TypeScript backend and a Kotlin/Compose Android application. The project is currently undergoing a significant architectural refactor. While the foundation (routing, types, environment configuration) is well-structured using modern tools like Hono and Zod, the implementation is currently at a "skeleton" stage. Core functionality—specifically the DSL engine and robust data persistence—is either missing or currently implemented as volatile in-memory storage.

## Critical Issues

### 1. Volatile Data Persistence (Mock Mode)
The server defaults to an in-memory `Map`-based storage when Supabase credentials are missing. This is intentional for local development but currently represents the *only* functional storage path as real DB initialization logic is placeholder. All user registrations, notifications, and sync states are lost on server restart.
- **Impact:** Total data loss.
- **Location:** `mr_notifier-server/src/core/db/database.ts`, `mr_notifier-server/src/modules/auth/auth.routes.ts`, `mr_notifier-server/src/modules/sync/sync.routes.ts`.

### 2. Missing Core Logic (DSL Engine)
The project structure contains directories for a DSL compiler, executor, lexer, and parser, but these are currently empty. The DSL engine is described as a "core feature" but does not yet exist in the source.
- **Impact:** System cannot process complex event rules as intended.
- **Location:** `mr_notifier-server/src/dsl/`.

### 3. Zero Test Coverage
Despite having `vitest` and `androidx.test` dependencies, there are no actual test files in the project.
- **Impact:** High risk of regressions during the ongoing refactoring; zero verification of logic correctness.
- **Location:** Entire repository.

### 4. Android Environment Constraints
The `REFACTORING_PLAN.md` notes that industry-standard libraries like **Room** and **kapt** were removed because they "don't work on Termux". This forces the app to use in-memory storage or simple preferences, which is not suitable for a production-grade notification manager.
- **Impact:** Compromised scalability and reliability of the mobile client.
- **Location:** `forked-version/REFACTORING_PLAN.md`.

## Major Issues

### 1. API Inconsistency (Client vs. Server)
The Android client (`forked-version`) expects endpoints for device registration, notification acknowledgement, and unread counts that are either missing or only exist as placeholders on the server.
- **Impact:** Client crashes or functional failure when interacting with the backend.
- **Location:** `forked-version/app/src/main/java/com/signalnest/app/data/api/SignalNestApi.kt` vs `mr_notifier-server/src/routes/`.

### 2. Naming Convention Mismatch
The codebase mixes `camelCase` (e.g., `userId`, `entityType` in mock objects) and `snake_case` (e.g., `user_id`, `entity_type` in Supabase queries and `entities.ts`). 
- **Impact:** Increased developer friction and potential runtime bugs during serialization/deserialization.
- **Location:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`.

### 3. Lack of Data Access Abstraction
The logic for switching between "Mock Mode" and "Supabase Mode" is implemented directly within the route handlers using `if (isMock())` blocks.
- **Impact:** Poor maintainability; violates the Single Responsibility Principle. Should be abstracted into a Repository pattern.
- **Location:** All files in `mr_notifier-server/src/modules/`.

## Minor Issues

### 1. Loose CORS Policy
The server uses `Access-Control-Allow-Origin: '*'.` 
- **Impact:** Security risk for web-based clients; should be restricted to known domains in production.
- **Location:** `mr_notifier-server/src/index.ts`.

### 2. Incomplete Synchronization Logic
The sync endpoint filters by `is_deleted = false` but provides no mechanism for the client to know which items *were* deleted since the last sync.
- **Impact:** Client local storage will become out of sync with the server over time.
- **Location:** `mr_notifier-server/src/modules/sync/sync.routes.ts`.

## Long-term Risks
- **Platform Lock-in (Termux):** Restricting the Android app's architecture to what runs on Termux will prevent the use of modern Android development tools (Hilt, Room, KSP), leading to a "legacy" codebase from the start.
- **Feature Creep vs. Implementation Gap:** The `entities.ts` file defines 20+ interfaces (Stars, Forks, PullRequests, DeliveryLogs) that have zero supporting logic. There is a risk of the project becoming over-engineered in design but under-implemented in reality.

## Suggested Roadmap Updates
1. **Immediate:** Implement SQLite-based file storage for the server's "Mock Mode" to prevent data loss during development.
2. **Structural:** Implement the Repository pattern in the backend to abstract database logic away from Hono routes.
3. **Validation:** Write unit tests for the Auth and Sync logic using Vitest.
4. **Core:** Implement a MVP version of the DSL lexer/parser to enable basic rule-based filtering.
5. **Client:** Investigate "Termux-friendly" ways to use Room or move development to a standard environment to unblock industry-standard Android practices.

## Score: 35/100
*The project has a solid architectural vision and clean initial setup, but the current implementation is mostly skeleton-code with broken persistence and no tests.*
