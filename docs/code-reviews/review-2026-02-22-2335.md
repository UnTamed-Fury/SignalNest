# Code Audit Report: SignalNest (mr-notifier)
**Date:** 2026-02-22 23:35
**Auditor:** Gemini CLI

---

## 1. Executive Summary
The project, now branded as **SignalNest**, is an event intelligence and notification system consisting of a Kotlin-based Android application and a TypeScript-based Hono server. The codebase demonstrates high structural integrity and follows modern architectural patterns (MVVM on Android, modular routes on Server). However, the core "Event Intelligence" feature (the DSL engine) is currently a collection of empty directories, and several key backend endpoints remain as placeholders. While the foundation is strong, the project is currently in a "heavy development" phase with significant missing functionality and some critical logic flaws in the backend.

## 2. Critical Issues
- **Missing Core Logic (DSL Engine):** The `src/dsl` directory in the server is entirely empty despite being a central value proposition of the system.
- **Backend Pagination Bug:** In `notificationRoutes.get('/')`, the `start` and `end` variables are used in the real Supabase path but are only defined within the `isMock()` block. This will cause a crash/error in production.
- **Incomplete Mock Database:** The `createMockSupabase` client is missing implementation for several methods (like `.range()`) that are actively used in the routes, leading to runtime failures in mock mode.

## 3. Major Issues
- **Missing Webhook Implementation:** The GitHub webhook endpoint is a placeholder. Without this, the system cannot ingest external events automatically.
- **In-Memory Storage Risk:** The mock mode uses in-memory storage for both users and notifications. On serverless platforms or during restarts, all local development data is lost.
- **Lack of Automated Tests:** Despite a `vitest` dependency and a `test` script, there are zero test files in the server or Android project.
- **Inconsistent Branding:** The repository contains two Android projects (`mr_notifier` and `forked-version`). While `forked-version` is the superior implementation, this duplication causes confusion and maintenance overhead.

## 4. Minor Issues
- **Hardcoded JWT Secret:** The `.env` file contains a hardcoded `JWT_SECRET`. While standard for local dev, it poses a risk if developers accidentally use it in production.
- **Hardcoded Sync State:** `syncRoutes.post('/')` hardcodes `has_more: false`, which prevents the system from handling large volumes of notifications through chunking.
- **Termux Constraints:** The project has dropped `Room` and `kapt` due to Termux compatibility issues, which limits local data persistence on the Android device.

## 5. Long-term Risks
- **Scalability of In-Memory Maps:** Using `Map` for mock storage will lead to memory exhaustion if many notifications are created during long-running dev sessions.
- **Maintenance of Custom DSL:** Building a custom lexer/parser/compiler for notification rules is a high-maintenance endeavor. Without proper documentation and tests, this will become a major point of failure.

## 6. Suggested Roadmap Updates
1.  **URGENT:** Fix the `start`/`end` variable scope bug in `notifications.routes.ts`.
2.  **Implementation:** Prioritize the DSL engine or integrate an existing rule engine (like json-logic or a lightweight scripting language).
3.  **Persistence:** Replace the in-memory mock store with a local SQLite file (using `better-sqlite3`) to prevent data loss during development.
4.  **Cleanup:** Deprecate and remove the `mr_notifier/` Android directory in favor of `forked-version/`.
5.  **Validation:** Implement basic unit tests for the authentication and notification logic.

## 7. Score (0–100)
**Score: 68/100**

*Rationale:* The architectural foundation is excellent (85+), but the current "missing" state of core features (DSL, webhooks) and the presence of critical scope bugs in the backend significantly pull down the score. Once the DSL is implemented and the pagination bugs are fixed, the score would easily jump to 85+.
