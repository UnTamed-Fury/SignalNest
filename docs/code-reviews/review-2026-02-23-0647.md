# Code Audit Report: SignalNest / MR Notifier
**Date:** 2026-02-23  
**Status:** Pre-Alpha / In-Transition  
**Auditor:** Gemini CLI

---

## 1. Executive Summary
The project is in a transitional state, refactoring from a basic notification tool to a more robust "Event Intelligence System" branded as SignalNest. While the architectural choice of Hono (Backend) and Jetpack Compose (Android) is modern and scalable, the current implementation is incomplete and contains critical security risks and logic flaws. The lack of automated tests and the presence of leaked credentials in configuration templates are the most immediate concerns.

---

## 2. Critical Issues
### 🔴 Security Risk: Leaked Credentials in Template
The `.env.template` file contains a specific, seemingly real `DATABASE_URL` with a password. This is a severe security risk and could lead to unauthorized database access if these credentials are valid or were ever used in production.
- **File:** `mr_notifier-server/.env.template`

### 🔴 Logic Bug: Undefined Variables in Notification Routes
The `GET /api/notifications` route uses `start` and `end` variables that are never defined, which will cause the server to crash when running in non-mock mode.
- **File:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`

### 🔴 Security Gap: Missing Authentication on API Routes
While the `auth` module exists, the notification and sync routes do not currently use any authentication middleware. Any user can access, create, or modify notifications if the server is exposed.
- **File:** `mr_notifier-server/src/index.ts`

---

## 3. Major Issues
### 🟠 Broken Sync Logic
The `sync` route incorrectly reads `currentVersion` from `mockSyncState` even when operating in real database mode. This will lead to inconsistent versioning across server restarts or multi-instance deployments.
- **File:** `mr_notifier-server/src/modules/sync/sync.routes.ts`

### 🟠 Absolute Lack of Automated Tests
There are **zero** project-specific unit or integration tests in the entire repository. This makes refactoring extremely dangerous and fails the "A-grade production standard" requirement.
- **Areas affected:** All

### 🟠 Split Logic Maintainability
Most routes use an `if (isMock())` pattern to branch between in-memory and Supabase logic. This duplicates business logic and significantly increases the chance of bugs, as seen in the sync and notification modules.
- **Suggested Fix:** Implement a Repository Pattern or Data Access Layer to abstract the storage mechanism.

---

## 4. Minor Issues & Technical Debt
- **Empty Modules:** Several planned features (DSL Engine, Adapters for GitHub/Discord) exist only as empty directory structures.
- **In-Memory Volatility:** The `mockUsers` and `mockNotifications` stores are in-memory and lost on every server restart, which hinders local development testing.
- **Basic Logging:** The file-based logger uses `fs.appendFile` directly, which may suffer from race conditions or performance bottlenecks under high concurrency.
- **Inconsistent Naming:** The project is referred to as both `mr_notifier` and `SignalNest` across different files.

---

## 5. Long-term Risks
- **Scalability:** The current sync logic is based on `created_at` timestamps, which does not handle hard deletes or updates to existing entities correctly. A version-based sync for all entities is required for robust multi-device synchronization.
- **Maintenance:** Without a test suite, the codebase will become unmanageable as complexity increases (especially when the DSL engine is implemented).
- **Termux Limitations:** The Android project is restricted by Termux (e.g., no Room/Kapt), which limits the ability to use standard Android libraries and might lead to "forked" development paths.

---

## 6. Suggested Roadmap Updates
1. **Immediate:** Sanitize `.env.template` and rotate any potentially compromised credentials.
2. **Immediate:** Fix the `start/end` undefined variables in `notifications.routes.ts`.
3. **Short-term:** Implement a JWT middleware and apply it to all `/api/*` routes.
4. **Short-term:** Create a base test suite using Vitest (Backend) and JUnit (Android).
5. **Short-term:** Refactor data access into a Repository pattern to remove `isMock()` branches from route handlers.
6. **Medium-term:** Implement the GitHub/Discord adapters to move beyond placeholder status.

---

*The project has a solid architectural vision and modern stack, but the current state is functionally broken in non-mock mode and carries significant security risks.*
