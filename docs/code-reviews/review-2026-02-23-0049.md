# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Version:** 3.0.0 (Server) / 1.0.0 (App)

---

## 1. Executive Summary
The SignalNest project remains in a fragile state. While the architectural foundations for a multi-platform notification system are present, the implementation is riddled with critical logic errors that prevent production readiness. Major security risks regarding JWT secret handling on the client side, along with several "production-mode" (Supabase) bugs that have likely been masked by the prevalence of "mock-mode" development, were identified. The core DSL engine feature is still entirely absent.

**Score: 35/100** (Decreased from previous review due to persistent critical bugs and discovery of new sync-meta logic flaws).

---

## 2. Critical Issues (P0)

### 2.1. Security: JWT Secret Disclosure (Android)
**Location:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`, `SettingsScreen.kt`
The application allows users to manually enter and save a `JWT_SECRET`. This secret is then stored in `DataStore`. This is a catastrophic security vulnerability. JWT secrets must remain on the server. If a client knows the secret, they can forge tokens for any user, completely bypassing authentication.

### 2.2. Logic: Authentication Function Reference Bug (Server)
**Location:** `mr_notifier-server/src/modules/auth/auth.routes.ts`
The login route checks `if (!isMock)` on line 185. Since `isMock` is an imported function, `!isMock` evaluates to `false` regardless of the configuration. This prevents the "Real Supabase" last-login update from ever executing.

### 2.3. Logic: Undefined Variables in Production Pagination (Server)
**Location:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`
The Supabase query block (lines 40-44) uses `.range(start, end)`, but `start` and `end` are only defined within the `isMock()` conditional block. In production (Supabase) mode, this will throw a `ReferenceError` and crash the endpoint.

### 2.4. Logic: Incorrect Sync Version Calculation (Server)
**Location:** `mr_notifier-server/src/modules/sync/sync.routes.ts`
In Supabase mode (line 69), the `currentVersion` is calculated using `mockSyncState.get(user_id)`. `mockSyncState` is an in-memory Map. This means that in a real production environment, the sync versioning will reset every time the server restarts, leading to massive data inconsistency and duplication on clients.

## 3. Major Issues (P1)

### 3.1. Persistence: Missing Room DB (Android)
**Location:** `forked-version/app/src/main/java/com/signalnest/app/data/repository/SignalNestRepository.kt`
The repository continues to use `MutableStateFlow` with in-memory Lists for all core entities (Notifications, Notes, Todos). All data is lost when the app process is terminated. For a notification aggregator, persistent local storage is a non-negotiable requirement.

### 3.2. Architecture: DSL Engine is Still a Skeleton
**Location:** `mr_notifier-server/src/dsl/`
Despite having a directory structure for Lexer, Parser, Validator, and Executor, all directories are empty. The core differentiating feature of SignalNest (DSL-driven event processing) exists only as a folder structure.

### 3.3. Permissions: Missing Internet Permission
**Location:** `mr_notifier/app/src/main/AndroidManifest.xml`
The primary Android application manifest is still missing the `<uses-permission android:name="android.permission.INTERNET" />`. While the `forked-version` may have it, the main project directory remains broken for network communication.

---

## 4. Minor Issues (P2)

### 4.1. Architecture: Hardcoded User Identity
The Android repository (`SignalNestRepository.kt`) continues to hardcode `userId = 0` for all sync and count operations. This limits the application to a single-user prototype model and conflicts with the multi-user server capabilities.

### 4.2. Inconsistent Source Versions
There is significant divergence between the `mr_notifier/` and `forked-version/` directories. Resources (like `ic_notification`) and logic (like `SyncWorker`) are present in one but missing or different in the other, creating a confusing development environment.

### 4.3. UI: Mock Mode UX
The `SettingsScreen.kt` provides "Quick Presets" for server URLs but doesn't validate if the server is actually reachable or if the provided JWT secret (which shouldn't be there) is even valid for the target server.

---

## 5. Long-term Risks
- **Data Integrity**: The current sync-meta logic (resetting on server restart) will cause clients to receive duplicate notifications or miss updates entirely once production database scaling is attempted.
- **Maintenance Nightmare**: Maintaining two separate, partially functional Android codebases will lead to "logical rot" where bugs fixed in one version persist in the other.
- **Production Readiness**: The project relies heavily on `mock-mode`. The critical bugs found in the Supabase code paths suggest that production-mode testing is currently non-existent in the CI/CD pipeline.

---

## 6. Suggested Roadmap Updates
1. **IMMEDIATE**: Remove the `JWT_SECRET` input from the Android app. Correct the `isMock()` calls in the server.
2. **Persistence**: Implement Room DB in the `forked-version` and make it the primary Android codebase.
3. **Database Fix**: Update `sync.routes.ts` to fetch the current sync version from the `sync_meta` table instead of an in-memory Map.
4. **DSL Implementation**: Prioritize the Lexer and Parser to allow basic rule processing.
5. **Unified Manifest**: Ensure the main Android manifest contains all necessary permissions (INTERNET, POST_NOTIFICATIONS).

---

## 7. Technical Debt Flag
- **Functional Inconsistency**: The use of function references (`isMock`) instead of calls (`isMock()`) in conditional logic is a recurring pattern of technical debt.
- **Ghost Features**: Multiple endpoints (e.g., GitHub webhooks) are placeholders that return 200 OK but perform no action, which can mislead external integrations.
