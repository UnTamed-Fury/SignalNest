# Code Review Report: SignalNest (mr_notifier) - 2026-02-23 04:15

## Executive Summary
SignalNest is in a transitional phase. While the repository contains a "legacy" app (`mr_notifier`) which is fundamentally broken, the "forked-version" demonstrates a significant architectural leap towards production standards using modern Android practices (Compose, WorkManager, Clean Architecture). However, the project remains a high-level prototype because the server-side DSL engine is entirely missing, and the Android application relies on volatile in-memory storage. A critical logic bug in the server's authentication module also prevents proper database integration during testing.

**Overall Grade: C- (Architectural Prototype)**
*Improvement from previous 'D' due to the superior 'forked-version' structure, though major implementation gaps remain.*

---

## Critical Issues (Blocking Production)

### 1. In-Memory Data Volatility (Android)
**File:** `forked-version/app/src/main/java/com/signalnest/app/data/repository/SignalNestRepository.kt`
The application uses `MutableStateFlow` in-memory storage for all notifications, notes, and todos.
- **Impact:** All user data, read statuses, and local modifications are lost whenever the Android OS kills the app process (common in background usage).
- **Requirement:** Implement a persistent database (Room) to ensure data survives process death.

### 2. Broken 'Mock Mode' Logic (Server)
**File:** `mr_notifier-server/src/modules/auth/auth.routes.ts`
The code uses `if (!isMock)` instead of `if (!isMock())`. Since `isMock` is a function, it is always truthy, meaning the negation is always false.
- **Impact:** The server *always* executes the mock logic for updating `last_login`, even when a real Supabase connection is configured. This prevents actual database updates for user sessions.
- **Requirement:** Change to `if (!isMock())`.

### 3. Missing Background Sync Initialization (Android)
**File:** `forked-version/app/src/main/java/com/signalnest/app/worker/SyncWorker.kt`
While a robust `SyncWorker` is implemented using `WorkManager`, the `schedulePeriodicSync` method is never called in `MainActivity` or `SignalNestApp`.
- **Impact:** Background synchronization will never occur; notifications only sync when the app is actively opened and the `HomeViewModel` is initialized.
- **Requirement:** Call `SyncWorker.schedulePeriodicSync(this)` in `SignalNestApp.onCreate()`.

## Major Issues

### 1. Core Engine Absence
**Directory:** `mr_notifier-server/src/dsl/`
The core value proposition—the SNRL DSL engine—remains unimplemented (empty directories).
- **Impact:** The system currently only functions as a simple notification proxy without any intelligence or event-processing capabilities.

### 2. Incomplete Database Initialization
**File:** `mr_notifier-server/src/core/db/database.ts`
The `initDatabase()` function still lacks the logic to apply `schema.sql`.
- **Impact:** Deployment requires manual SQL execution, increasing the risk of environment drift and setup friction.

### 3. Zero Test Coverage
There are no unit, integration, or UI tests in either the server or the Android application.
- **Requirement:** Implement Vitest for the server and JUnit/Compose-Test for the Android app, focusing on the Sync logic and Auth flows.

---

## Minor Issues & Technical Debt

- **Unused Legacy Code**: The `mr_notifier` directory contains a non-functional version of the app (missing permissions, broken scheduling). It should be archived or removed to reduce cognitive load.
- **Sync Logic Race Condition**: The `syncNotifications` logic in `SignalNestRepository` performs a naive merge that may overwrite local state if the server's response is slightly stale.
- **Missing Endpoints**: The Android `SignalNestApi` interface defines endpoints (like `/api/devices/register`) that the server hasn't implemented yet.
- **Termux Constraints**: The use of in-memory storage is justified in comments by "kapt limitations on Termux," which represents a significant platform-specific bottleneck for development.

---

## Long-term Risks

- **Platform Dependency**: Relying on Termux-compatible build tools (avoiding `kapt`) significantly limits the use of industry-standard libraries like Room or Hilt, leading to "reinventing the wheel" for dependency injection and persistence.
- **State Synchronization**: As the app moves toward persistence, the complexity of syncing local vs. remote state (especially with "unread" and "pinned" flags) will increase significantly without a robust versioning strategy.

---

## Suggested Roadmap Updates

1.  **Persistence First**: Prioritize Room database implementation, even if it requires switching build environments (e.g., GitHub Actions) to support KSP/Kapt.
2.  **Server Integrity**: Fix the `isMock()` bug and implement the `initDatabase` schema runner.
3.  **Sync Reliability**: Initialize `SyncWorker` on app startup and implement a "pull-to-refresh" feedback loop in the UI.
4.  **DSL Scaffolding**: Implement a basic "passthrough" executor for the DSL to establish the processing pipeline.
