# Code Audit Report: Mr. Notifier (SignalNest)

**Date:** 2026-02-23 01:50 UTC
**Auditor:** Gemini CLI
**Target:** `mr_notifier-server` (Backend) & `forked-version` (Android Client)

## 1. Executive Summary

The **Mr. Notifier (SignalNest)** project is a prototype-stage notification management system consisting of a Node.js/Hono backend and an Android Jetpack Compose client.

**Status:** ⚠️ **Alpha / Prototype**
**Score:** 35/100

While the project utilizes a modern technology stack (Hono, Supabase, Jetpack Compose), it fails to meet production standards due to critical gaps in security, data persistence, and core functionality. The "DSL Engine" advertised in the project description is non-existent (empty directories). The application currently operates on volatile in-memory storage for both client and server (in default mock mode), meaning all data is lost upon restart. Authentication exists but is bypassed by the core business logic.

## 2. Critical Issues (P0)

### 2.1 Security: Unprotected API Endpoints
- **Location:** `mr_notifier-server/src/index.ts`, `notifications.routes.ts`, `sync.routes.ts`
- **Issue:** The API routes `/api/notifications` and `/api/sync` are completely public. They do not implement any JWT verification middleware.
- **Impact:** Any user can query, create, or modify notifications for ANY user by simply changing the `user_id` query parameter (e.g., `?user_id=1`).
- **Evidence:** `notificationRoutes.get('/', ...)` accesses `c.req.query('user_id')` without validation against an authenticated token.

### 2.2 Data Integrity: Volatile Storage
- **Location:** `forked-version/.../SignalNestRepository.kt` & `mr_notifier-server/.../mock-store.ts`
- **Issue:**
    - **Client:** Uses `MutableStateFlow` as the primary data source. There is no Room database or SharedPreferences for notification persistence.
    - **Server:** Defaults to "Mock Mode" which uses in-memory `Map` objects.
- **Impact:** All user data (notifications, read status, pins) is strictly temporary and vanishes when the app is closed or the server restarts.

### 2.3 Architecture: Missing "DSL Engine"
- **Location:** `mr_notifier-server/src/dsl/`
- **Issue:** The `lexer`, `parser`, `executor`, and `validator` directories are empty.
- **Impact:** A core feature advertised in `package.json` ("DSL Engine") is unimplemented.

### 2.4 Data Sync: Destructive Logic
- **Location:** `SignalNestRepository.kt` (Client)
- **Issue:** The `syncNotifications` function implements a "last-write-wins" strategy that effectively overwrites local state with server state.
    - **Pinning/Deletion:** `togglePin` and `deleteNotification` are local-only operations. They are not synced to the server.
    - **Result:** When `sync` runs, it fetches the server's version (unpinned/undeleted) and overwrites the local changes, causing pinned items to unpin and deleted items to reappear.

## 3. Major Issues (P1)

### 3.1 Performance: N+1 API Calls
- **Location:** `HomeViewModel.kt` -> `markAllAsRead`
- **Issue:** The client iterates through the local list and triggers a separate `markAsRead` API call for *each* notification.
- **Impact:** Massive network congestion and rate limiting issues for users with many notifications.

### 3.2 Server: Fragmented Mock State
- **Location:** `mr_notifier-server`
- **Issue:**
    - `auth.routes.ts` uses a local `mockUsers` map.
    - `notifications.routes.ts` uses `mockNotifications` from `mock-store.ts`.
    - `database.ts` uses a local `mockData` map.
- **Impact:** Data inconsistency. A user created via `/auth` does not exist in the context of the generic database mock, leading to potential reference errors or "ghost" data.

### 3.3 Client: Hardcoded User Identity
- **Location:** `SignalNestRepository.kt`
- **Issue:** API calls hardcode `userId = 0` (e.g., `api.sync(SyncRequest(userId = 0))`).
- **Impact:** The client is effectively hardwired to a "guest" or "admin" mode, ignoring the actual logged-in user's credentials.

## 4. Minor Issues (P2)

- **Style/Conventions:** Inconsistent naming in API (mix of `camelCase` and `snake_case`).
- **Search:** Client-side search is limited to the in-memory list; it does not query the server.
- **Placeholders:** Many server routes (e.g., GitHub webhooks) return "placeholder" JSON responses.
- **Documentation:** The project claims features (DSL) that do not exist.

## 6. Recommendations & Roadmap

### Immediate Actions (Next 24 Hours)
1.  **Secure Endpoints:** Implement a JWT middleware in Hono and apply it to `/api/*` routes. Verify `c.req.user.id` matches the requested resource owner.
2.  **Persist Client Data:** Replace `MutableStateFlow` in `SignalNestRepository` with a **Room Database**.
3.  **Fix Sync:**
    - Update server to accept `last_updated_at` timestamp.
    - Implement `DELETE /api/notifications/:id` and `PATCH /api/notifications/:id` (for pin status) on the server.
    - Update client to send local changes to server before fetching updates.

### Short Term (1-2 Weeks)
1.  **Implement Server Persistence:** Ensure `Supabase` connection is robust or replace `MockStore` with a local SQLite/PostgreSQL instance for dev.
2.  **Optimize Network:** Replace `markAllAsRead` loop with a single batch API call.
3.  **Implement DSL:** Either build the DSL engine or remove the empty directories/claims to reflect reality.

### Long Term
1.  **Push Notifications:** Integrate Firebase Cloud Messaging (FCM) properly (currently just a dependency).
2.  **Background Sync:** Utilize Android `WorkManager` for reliable background synchronization.
