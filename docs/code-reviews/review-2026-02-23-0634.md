# Static Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23 06:45 UTC
**Auditor:** Gemini CLI (Scheduled Audit)
**Version:** 3.0.0-beta

## Executive Summary
SignalNest is a notification aggregation system consisting of a Hono-based backend and a Kotlin/Jetpack Compose Android application. While the project architecture is clean and follows modern patterns (MVVM on Android, Modular routes on Backend), it currently suffers from **critical security vulnerabilities** and a complete **lack of automated testing**. The backend's reliance on a dual-mode (Mock vs. Supabase) implementation has introduced significant logic bugs in the production path. The project is currently at a "Prototype" stage and is not ready for production use.

## Critical Issues
### 1. Missing Authentication Middleware (Security)
**Location:** `mr_notifier-server/src/index.ts` and `mr_notifier-server/src/modules/notifications/`
**Description:** API routes under `/api/notifications` and `/api/sync` are completely unprotected. While the `/auth/me` endpoint verifies JWT tokens, no middleware is applied to the notification endpoints.
**Impact:** Any user can access, modify, or create notifications for any other user by simply passing a `user_id` in the query string or request body. This is a massive data breach risk.
**Risk Level:** CRITICAL

### 2. Production ReferenceError in Notification Listing (Logic)
**Location:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`
**Description:** In the `GET /api/notifications` route, the variables `start` and `end` used for pagination in the "Real Supabase mode" are defined only inside the `if (isMock())` block.
**Impact:** The server will crash with a `ReferenceError` when running in production mode (with Supabase) when trying to list notifications.
**Risk Level:** CRITICAL

## Major Issues
### 1. Absolute Lack of Tests (Reliability)
**Location:** Entire repository.
**Description:** There are zero project-specific unit, integration, or UI tests. The only tests found were in `node_modules`.
**Impact:** Impossible to verify correctness or prevent regressions. The project does not meet any professional engineering standard for reliability.
**Risk Level:** MAJOR

### 2. Lack of Persistent Storage on Android (Architecture)
**Location:** `forked-version/app/src/main/java/com/signalnest/app/data/local/LocalData.kt`
**Description:** The Android app currently uses in-memory `StateFlow` storage instead of a persistent database like Room.
**Impact:** All user data (notifications, notes, todos) is lost every time the app process is killed or the device is restarted. This violates the "offline-first" goal stated in the roadmap.
**Risk Level:** MAJOR

### 3. Insecure User Identification (Security)
**Location:** Multiple routes in `notificationRoutes.ts` and `sync.routes.ts`.
**Description:** The system relies on `user_id` passed from the client (query param or body) to identify the owner of the data, rather than deriving it from a verified JWT token.
**Impact:** Even if JWT middleware is added, if the code continues to use the client-provided `user_id`, IDOR (Insecure Direct Object Reference) vulnerabilities will persist.
**Risk Level:** MAJOR

## Minor Issues
### 1. Misleading Package Description
**Location:** `mr_notifier-server/package.json`
**Description:** Description claims "Event Intelligence System with DSL Engine", but all `dsl` directories are empty.
**Impact:** Misleading for developers and stakeholders.

### 2. Version Confusion
**Location:** Root directory contains `mr_notifier` and `forked-version`.
**Description:** Having two Android projects in the same repo without clear deprecation headers causes confusion. `forked-version` appears to be the active one.

### 3. Code Duplication in Backend
**Location:** `mr_notifier-server/src/modules/`
**Description:** Every route contains a large `if (isMock())` block that duplicates logic instead of using a polymorphic database adapter or a more robust mock client.

## Long-term Risks
1. **Scalability:** The current sync logic (`last_sync_timestamp`) is primitive and may lead to missed events if multiple events have the exact same timestamp or if clock skew occurs.
2. **Maintenance:** The divergence between Mock and Production code paths will lead to "it works on my machine" bugs (as seen with the `start`/`end` bug).
3. **Security:** Lack of rate limiting on auth endpoints makes the system vulnerable to brute-force attacks.

## Suggested Roadmap Updates
1. **Immediate (Next 24h):** Implement JWT Auth Middleware and apply to all `/api/*` routes.
2. **Immediate (Next 24h):** Fix the `ReferenceError` in `notifications.routes.ts`.
3. **High Priority:** Implement a minimal Room database on Android to enable persistence.
4. **High Priority:** Add Vitest for backend and JUnit/Espresso for Android. At least 40% coverage for core logic.
5. **Architectural:** Refactor `database.ts` to provide a truly transparent mock that doesn't require `if (isMock())` in every route.
