# Code Audit Report: SignalNest (mr-notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI
**Overall Score: 45/100**

## 1. Executive Summary
The SignalNest project (formerly mr-notifier) is in a transitional state, likely undergoing a major refactor (v2 to v3). While the database schema and data models are well-architected and "A-grade," the current server implementation is significantly lagging behind, consisting mostly of placeholder routes and a primitive mock database layer. There are critical security vulnerabilities related to authentication and secret management that must be addressed before any production deployment.

## 2. Critical Issues
- **Missing Authentication Middleware:** Core API endpoints (`/api/notifications`, `/api/sync`) lack authentication. Any user can access or manipulate data for any `userId` by simply passing it as a query parameter or in the request body.
- **JWT Secret Leak Risk:** The Android application's `ApiClient` and `DataStore` include logic for handling a `JWT_SECRET`. Clients should *never* have access to the server's signing secret, as this allows them to forge valid authentication tokens.
- **Mock Logic in Production Paths:** The server code is littered with `if (isMock())` blocks. This logic should be abstracted into a repository pattern or separate data access layer to prevent dev-only logic from leaking into and potentially breaking production flows.

## 3. Major Issues
- **Placeholder Implementation:** Key features like the DSL engine (compiler, lexer, parser) and GitHub webhooks are currently empty directories or placeholder routes, despite being core to the project's value proposition.
- **Inconsistent Naming Conventions:** The project mixes `snake_case` (database/schema) and `camelCase` (some API fields) inconsistently across the stack, which will lead to integration bugs.
- **In-House Supabase Mock:** The custom mock Supabase client is very basic and does not accurately reflect Supabase behavior (e.g., O(N) filtering in-memory). This creates a "works on my machine" risk where code behaves differently in dev vs. prod.

## 4. Minor Issues
- **Duplicate Dependencies:** `androidx.datastore:datastore-preferences` is declared twice in `forked-version/app/build.gradle.kts`.
- **Hardcoded Localhost:** The Android app has hardcoded `10.0.2.2:3000` for the emulator, which is fine for dev but needs a proper flavor/build config system for production.
- **Missing Error Handling:** Many catch blocks use generic `Error` types and simply log them, returning 500 without structured error codes for the client to handle.

## 5. Architectural Weaknesses & Long-term Risks
- **Coupling to Supabase:** While Supabase is a great tool, the code is tightly coupled to it. Abstracting the database layer would improve testability and future-proof the project.
- **Scalability of Sync:** The current sync logic relies on a simple timestamp/version check. As the number of notifications grows, the O(N) filtering in the mock layer and the lack of robust indexing on the `sync_version` in the real DB (though indices exist) might become a bottleneck.
- **Maintenance Burden of Custom DSL:** Building a custom DSL engine from scratch (lexer/parser/compiler) is a high-maintenance task. Using an existing expression language or a lighter-weight rule engine might be more sustainable.

## 6. Suggested Roadmap Updates
1.  **URGENT:** Implement global authentication middleware and enforce user isolation on all `/api/*` routes.
2.  **URGENT:** Remove any logic that shares the `JWT_SECRET` with the client.
3.  **Refactor Data Access:** Move mock logic out of route handlers and into a dedicated `Repository` layer.
4.  **Implement DSL Engine:** Prioritize the lexer and parser for the DSL as it is the project's defining feature.
5.  **Synchronization:** Align the Android `SignalNestApi` interface with the actual server implementation.

## 7. Technical Debt Flagged
- Empty DSL directories.
- Placeholder routes for Webhooks and Sync.
- Mixed usage of `any` in TypeScript.
- Lack of unit tests for the server routes and DSL logic.

## 8. Final Score: 45/100
*The project has a solid structural foundation (Schema/Models) but the current implementation is too incomplete and insecure for a production standard.*
