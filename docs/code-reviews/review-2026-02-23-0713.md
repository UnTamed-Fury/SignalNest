# SignalNest / MR Notifier - Static Code Audit Report

**Date:** 2026-02-23  
**Audit Duration:** 10 minutes  
**Target:** Full Repository (`mr_notifier-server`, `mr_notifier`, `forked-version`)  
**Score:** 45/100

---

## Executive Summary

The project is currently in a "Transitional/Refactoring" state. The core backend (`mr_notifier-server`) is built on Hono/Node.js and intended to work with Supabase, but defaults to an in-memory mock mode. The Android side is fragmented between a more complete `forked-version` and a skeletal `mr_notifier` refactor. While the architectural vision (canonical trees, versioned sync, custom DSL) is ambitious and well-defined, the implementation is riddled with critical logic flaws, missing core components (DSL engine, webhooks), and significant technical debt regarding data persistence.

---

## Critical Issues

### 1. Backend Logic Bug: Authentication Mock Check
In `mr_notifier-server/src/modules/auth/auth.routes.ts`, the code checks `if (!isMock)` instead of calling the function `if (!isMock())`. Since `isMock` is a function reference, it always evaluates to truthy, preventing the database update of `last_login` in production mode.
- **Location:** `auth.routes.ts:182`
- **Impact:** `last_login` is never updated in real Supabase mode.

### 2. Backend Logic Bug: Undefined Variables in Notification List
In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the pagination query uses variables `start` and `end` which are never declared or calculated.
- **Location:** `notifications.routes.ts:37`
- **Impact:** The notification list endpoint will crash or fail to execute correctly when using real Supabase mode.

### 3. Flawed Sync Mechanism
The synchronization logic in `sync.routes.ts` relies on filtering by `created_at > last_sync_timestamp`. This is inherently broken for a notification system where existing notifications can be updated (e.g., marked as read). 
- **Impact:** Users will not see "Read/Unread" status updates synced to their devices if the notification was created before their last sync.
- **Technical Debt:** The `sync_version` field in the schema is ignored in the actual sync query.

## Major Issues

### 1. High Risk of Data Loss (In-Memory Reliance)
- **Backend:** Defaults to `mock-store.ts` (Map-based) if `.env` is incomplete. All user data, notifications, and sync states are lost on server restart.
- **Android:** The `REFACTORING_PLAN.md` explicitly notes that Room database is disabled due to Termux/kapt limitations. The `SignalNestRepository` uses `MutableStateFlow` for all storage. 
- **Impact:** App data is lost every time the process is killed by the OS. This fails the "A-grade production standard" for a utility app.

### 2. Sync Performance Bottleneck
The Android `SignalNestRepository.syncNotifications()` call sends a `SyncRequest(userId = 0)` without a timestamp or version.
- **Impact:** The app fetches the *entire* notification history on every sync operation. This will lead to massive battery drain, data usage, and UI lag as the user's notification count grows.

### 3. Project Fragmentation
Having both `forked-version` and `mr_notifier` in the root creates confusion. `forked-version` is the functional codebase, but the "new" `mr_notifier` app is missing critical permissions like `INTERNET` and lacks any networking implementation (Retrofit/OkHttp).

---

## Minor Issues

### 1. Missing Core Features
- The `dsl/` directories are empty. The core "DSL-based notification" feature appears to be non-existent in the current implementation.
- GitHub webhooks are currently just placeholder strings.

### 2. Inconsistent Naming & Package Structure
Package names between the two Android versions (`com.signalnest.app` vs `fury.notifier`) and inconsistencies in `Type.kt` imports indicate a messy refactoring process.

---

## Long-term Risks

1. **Scalability:** The current sync model (full fetch) will break once a user reaches >1000 notifications.
2. **Maintenance:** The reliance on custom mock stores instead of a unified repository/abstraction layer makes it difficult to switch between local development and production safely.
3. **Security:** The `JWT_SECRET` in `.env.template` is visible, and while it's a template, the logic for "mock mode" bypasses many production security checks.

---

## Suggested Roadmap Updates

1. **Fix the Pagination/Mock Bugs:** Immediately correct the `isMock()` calls and calculate `start/end` for pagination.
2. **Implement Persistent Local Storage:** If Room is incompatible with the environment, use a simpler persistent store like **SQLDelight** or even a raw **SQLiteOpenHelper**. In-memory only is not viable.
3. **True Incremental Sync:** Update the `sync` endpoint to filter by `sync_version > client_version` and ensure *any* update (read status, delete) increments the `sync_version`.
4. **Consolidate Project:** Merge `forked-version` into the root or properly move to the new structure. Add `INTERNET` permission to the new manifest.
5. **Implement the DSL Engine:** This is the project's USP (Unique Selling Point); it should be prioritized over placeholder routes.
