# Code Audit Report: SignalNest (formerly MR Notifier)
**Date:** 2026-02-23
**Auditor:** Gemini CLI

## 1. Executive Summary
The project is currently in the early-to-mid stages of development (Phase 2 according to the roadmap). It is transitioning from a simpler "MR Notifier" concept to a more robust "SignalNest" event intelligence system. The architecture is modern, using Hono/Supabase for the backend and Jetpack Compose for the Android client. However, there is a significant gap between the intended feature set (DSL engine, rule processing) and the actual implementation. The most critical finding is the **complete absence of automated tests**, which represents a severe risk for a system intended to be a "Zero-loss" notification aggregator.

## 2. Critical Issues
| Issue | Description | Impact |
|-------|-------------|--------|
| **Zero Test Coverage** | No unit, integration, or E2E tests were found in either the backend or the Android repository. | High risk of regressions; unverifiable system behavior. |
| **Missing Core Logic** | The DSL engine (lexer, parser, compiler, executor) directories are empty, despite being marketed in `package.json` and having configuration constants. | Misleading project state; core "intelligence" feature is missing. |

## 3. Major Issues
| Issue | Description | Impact |
|-------|-------------|--------|
| **In-Memory App Storage** | Due to build issues (kapt/Room on Termux), the Android app uses in-memory storage. | **Critical for UX:** All notifications, notes, and local state are lost when the app process is killed. |
| **Naive Sync Logic** | The synchronization mechanism relies on a simple `gt(created_at, last_sync_timestamp)` query. | Fails to handle deletions, updates to existing notifications, or clock skew issues. |
| **Project Fragmentation** | The repository contains `mr_notifier`, `forked-version`, and `mr_notifier-server`. `forked-version` appears to be the "active" app, but this structure is confusing and cluttered. | Poor maintainability and onboarding experience. |
| **Mock-Heavy Development** | Extensive use of "mock mode" in the backend without a clear path to parity with the Supabase implementation. | Risk of "it works on my machine" bugs that only appear in production. |

## 4. Minor Issues / Technical Debt
- **Placeholder Endpoints:** GitHub webhooks and several API routes are currently placeholders.
- **Inconsistent Naming:** Mix of `mr_notifier` and `SignalNest` across the codebase.
- **Manual Auth Checks:** Auth is handled manually in some routes rather than through a global Hono middleware/guard (though some logic is shared).
- **Log File Handling:** The custom logger imports `fs/promises` dynamically inside the log function, which is slightly inefficient.

## 5. Architectural Evaluation
- **Scalability:** The Hono/Supabase stack is highly scalable. The move to a stateless architecture (as noted in global context) is a positive step.
- **Maintainability:** The modular structure of the backend (`modules/`, `core/`, `dsl/`) is good, but the empty directories and fragmented app versions hinder maintainability.
- **Security:** Use of `bcryptjs`, `jsonwebtoken`, and `zod` validation is a strong foundation. However, without tests, the security implementation cannot be verified.

## 6. Long-term Risks
1. **The "Feature Gap":** The project risks becoming "yet another notification app" if the promised DSL and intelligence engine are not implemented soon.
2. **Synchronization Reliability:** As the data grows, the naive sync will become a bottleneck and a source of data loss/inconsistency.
3. **Android Fragmentation:** Maintaining multiple versions of the Android app (`mr_notifier` vs `forked-version`) will drain resources.

## 7. Suggested Roadmap Updates
1. **TESTING (Immediate):** Implement a testing framework (Vitest for backend, JUnit/Espresso for Android) and reach at least 50% coverage of core logic.
2. **Consolidation:** Merge the "forked-version" into a single canonical Android project and remove legacy code.
3. **Advanced Sync:** Implement a proper sync protocol (e.g., using sequence numbers or vector clocks) that handles deletions and updates.
4. **DSL MVP:** Implement a minimal version of the DSL engine to move it out of "Long-term Expansion" and into "In Progress".

## 8. Score: 45/100
*The project has a good choice of tech stack and a clear vision, but the complete lack of tests and the presence of significant "dead code" (empty DSL folders) significantly lower the score for a production-standard assessment.*
