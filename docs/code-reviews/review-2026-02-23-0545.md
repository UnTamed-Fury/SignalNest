# Code Review: SignalNest (mr_notifier) - 2026-02-23

## Executive Summary
SignalNest is a modern notification and event intelligence system. The project currently consists of a functional TypeScript/Hono backend and a feature-rich Android app (v2.4.0 in `forked-version`). The architecture is sound, utilizing Supabase for data and Hono for a lightweight server. However, the project is in an early-to-mid stage: core features like the DSL Engine and external adapters (GitHub/Discord) are still pending. The most critical weakness is the **complete absence of automated tests** and a significant security risk in the notification retrieval endpoint.

---

## Critical Issues
| Issue | Description | Impact |
|-------|-------------|--------|
| **Missing Authentication Enforcement** | `GET /api/notifications` and `GET /api/sync/status` use `user_id` from query parameters without verifying it against the JWT `userId`. | High: Any user can view any other user's notifications by guessing their ID. |
| **Zero Test Coverage** | No unit, integration, or E2E tests were found in either the server or the Android app. | High: High risk of regressions during Phase 2 (DSL Engine) implementation. |

## Major Issues
| Issue | Description | Impact |
|-------|-------------|--------|
| **In-Memory Storage (App)** | Local persistence (Room) is "On Hold" due to Termux limitations. The app uses in-memory storage. | Medium: Data loss on app restart/process death. Limits offline utility. |
| **Sync Logic Limitations** | Sync is based on `created_at > last_sync_timestamp`. It does not handle deletions or updates to existing notifications (sync payloads don't propagate deletions). | Medium: Local database will become inconsistent with the server over time. |
| **Missing DSL Engine** | The core value proposition (SNRL) is not yet implemented; directories are empty. | Medium: The project currently functions as a simple notification relay, not an intelligence system. |

## Minor Issues
| Issue | Description | Impact |
|-------|-------------|--------|
| **Mock Data Volatility** | The server's mock mode uses in-memory Maps that reset on every restart. | Low: Inconvenient for development. |
| **Duplicate Codebases** | `mr_notifier/` and `forked-version/` contain different versions of the app, which may cause confusion. | Low: Maintainability overhead. |
| **Placeholder Adapters** | GitHub/Discord webhook endpoints are currently just returning placeholders. | Low: Development status (Phase 3 pending). |

---

## Long-term Risks
1. **Scalability of Sync:** As notification volume grows, timestamp-based sync without pagination or delta-encoding will become slow and heavy on bandwidth.
2. **Technical Debt (Termux):** Avoiding Room/Kapt due to Termux limitations forces the app to deviate from Android best practices, which will make migration to a "proper" build environment more difficult.
3. **DSL Complexity:** Implementing a custom lexer/parser/executor (Phase 2) is a high-effort task that can introduce significant security risks (e.g., infinite loops, memory exhaustion) if the execution engine isn't properly sandboxed.

---

## Suggested Roadmap Updates
1. **Immediate:** Implement an `authMiddleware` to secure all `/api` routes by extracting and verifying the JWT `userId`.
2. **Short-term:** Initialize `vitest` and `junit` with basic unit tests for the Auth and Notification logic.
3. **Short-term:** Resolve the Room database issue by using a non-Kapt alternative (e.g., SQLDelight) or moving the build to a standard CI/CD environment.
4. **Medium-term:** Prioritize the DSL Engine (Phase 2) to unlock the "Event Intelligence" features.

---

## Score: 68/100
*The codebase is well-written and structured, but the lack of testing, security gaps in API routes, and missing core features (DSL) prevent it from reaching an A-grade production standard.*
