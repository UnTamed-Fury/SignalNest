# Code Review Report: Mr. Notifier / SignalNest

**Date:** 2026-02-22 23:33 UTC
**Reviewer:** Gemini CLI
**Target:** `mr_notifier-server` (Backend) and `forked-version` (Android App)

## 1. Executive Summary

The project "SignalNest" (formerly Mr. Notifier) consists of a Node.js/Hono backend and an Android/Kotlin client. The architecture follows modern patterns (MVVM for Android, Modular Monolith for Backend), but the implementation is currently in a **early prototype stage**.

**Key Findings:**
-   **Core Feature Missing:** The "Event Intelligence" DSL (Domain Specific Language) engine, which is the project's primary value proposition, is completely missing from the implementation (empty directories).
-   **Data Persistence:** The Android app relies entirely on in-memory storage. All data (notifications, read states) is lost when the app process dies.
-   **Security Risks:** There are critical security vulnerabilities in both the backend (missing auth middleware, IDOR) and frontend (insecure token storage).
-   **Production Readiness:** The project is **not production-ready**. It scores effectively as a Proof of Concept (PoC).

**Score:** 35/100

---

## 2. Critical Issues (Must Fix)

### Backend (`mr_notifier-server`)
1.  **Missing DSL Implementation:**
    -   **Location:** `src/dsl/` and `src/modules/dsl/`
    -   **Issue:** The directories for `lexer`, `parser`, and `compiler` are empty. The core business logic for processing event rules does not exist.
2.  **Insecure API Endpoints (IDOR):**
    -   **Location:** `src/modules/notifications/notifications.routes.ts`
    -   **Issue:** Endpoints like `GET /` and `PATCH /mark-all-read` accept `user_id` as a query parameter without validating if the authenticated user matches that ID. A malicious user can read/modify another user's notifications.
3.  **Missing Authentication Middleware:**
    -   **Location:** `src/modules/notifications/notifications.routes.ts`
    -   **Issue:** The notification routes do not appear to use the auth middleware found in `auth.routes.ts`. They rely on the client passing the correct ID.
4.  **Broad CORS Configuration:**
    -   **Location:** `src/index.ts`
    -   **Issue:** `Access-Control-Allow-Origin: *` allows any website to make requests to the API, which is risky for an authenticated user-facing application.

### Android (`forked-version`)
1.  **Data Loss (Missing Persistence):**
    -   **Location:** `SignalNestRepository.kt`
    -   **Issue:** `_notifications` is stored in a `MutableStateFlow` (memory only). There is no local database (Room/SQLite). All local state is lost on restart.
2.  **Hardcoded Dev Configuration:**
    -   **Location:** `ApiClient.kt`
    -   **Issue:** Base URL is hardcoded to `http://10.0.2.2:3000/`. The app will not work on a physical device or against a production server without recompilation.
3.  **Insecure Token Storage:**
    -   **Location:** `ApiClient.kt`
    -   **Issue:** Auth token is stored in a static variable `authToken`. It is not persisted securely (e.g., `EncryptedSharedPreferences`). Users will be logged out unexpectedly.

---

## 3. Major Issues (Should Fix)

-   **Backend:**
    -   **Mock Logic in Production:** `if (isMock())` branches are scattered throughout controllers. This should be abstracted behind a `Repository` interface to keep controllers clean.
    -   **Logging:** Inconsistent use of `console.log` vs the structured `logger`.

-   **Android:**
    -   **Static Mutable State:** `ApiClient` is a singleton object with mutable state, making testing and thread-safety difficult.
    -   **JWT Secret Handling:** `setJwtSecret` suggests the client might be aware of the signing secret, which is a violation of the JWT security model (client should only hold the token).

---

## 4. Minor Issues (Polishing)

-   **Android:**
    -   Duplicate dependency declaration for `datastore-preferences` in `build.gradle.kts`.
    -   Placeholder screens (Sources, Schemas, Logs) need implementation.
-   **Backend:**
    -   `package.json` scripts use `tsx` (good for dev) but production build workflow isn't fully clear (just `tsc`).

---

## 5. Roadmap Recommendations

1.  **Phase 1: Security & Persistence (Immediate)**
    -   Implement Room Database in Android app to cache notifications.
    -   Secure the backend: Add Auth Middleware to all `/api/*` routes. Remove `user_id` from query params; derive it from the JWT token.
    -   Use `EncryptedSharedPreferences` for token storage on Android.

2.  **Phase 2: Core Functionality**
    -   Implement the DSL Engine. Start with a simple JSON-based rule processor if a full compiler is too complex for now.
    -   Implement the "Sources" module to actually ingest events (GitHub/Webhooks).

3.  **Phase 3: Cleanup**
    -   Remove Mock logic from controllers; use Dependency Injection to swap between Mock/Real repositories.
    -   Externalize configuration (Base URL) in the Android app (e.g., build config fields).
