# Code Audit Report: SignalNest / mr_notifier
**Date:** 2026-02-23 01:25 UTC  
**Scope:** Full Repository (Server + Android App)  
**Status:** ⚠️ High Risk / Production-Ready (Functional but Insecure)

## Executive Summary
The SignalNest project is a functional "Event Intelligence System" currently undergoing heavy refactoring. While the project achieves its core functional goals (notification scheduling, syncing, and state management), it currently sacrifices security for developer convenience (no-login mode). The architecture is modular but contains significant logic flaws in the synchronization engine and critical security gaps in the API layer.

## Critical Issues

### 1. Authentication Bypass (Server)
**Description:** The notification and sync endpoints (`/api/notifications`, `/api/sync`) are registered in `index.ts` without any authentication middleware.
**Impact:** Any user can read, create, or modify notifications if they know or guess a `user_id`. In "no-login" mode (default `user_id=0`), all data is effectively public.
**File:** `mr_notifier-server/src/index.ts`, `mr_notifier-server/src/modules/notifications/notifications.routes.ts`

### 2. Broken Sync Logic in Production Mode (Server)
**Description:** In `sync.routes.ts`, the "Real Supabase mode" logic incorrectly uses `mockSyncState.get(user_id)` to calculate the `currentVersion`. 
**Impact:** Synchronization versions will be inconsistent and volatile in production, as `mockSyncState` is an in-memory Map that resets on server restarts or lambda cold starts.
**File:** `mr_notifier-server/src/modules/sync/sync.routes.ts`

### 3. Insecure Multi-tenancy (Server)
**Description:** Endpoints accept `user_id` in the request body or query params but do not verify this ID against the JWT owner.
**Impact:** An authenticated user can access any other user's data by simply changing the `user_id` parameter.
**File:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, `mr_notifier-server/src/modules/sync/sync.routes.ts`

### 4. Permissive CORS Policy
**Description:** `Access-Control-Allow-Origin` is set to `*`.
**Impact:** Increases vulnerability to CSRF and unauthorized cross-origin data access if the API is exposed publicly.
**File:** `mr_notifier-server/src/index.ts`

## Major Issues

### 1. Technical Debt: In-Memory Storage (Android)
**Description:** The Android app uses in-memory storage for notifications, notes, and todos because "kapt crashes on Termux".
**Impact:** All local state (including pins and local notes) is lost when the app process is killed.
**File:** `forked-version/app/src/main/java/com/signalnest/app/data/repository/SignalNestRepository.kt`

### 2. Thread Safety & Initialization Bugs (Android)
**Description:** `ApiClient` uses static variables for `authToken` and `baseUrl`. The `api` instance is `lazy` and initialized with the default `baseUrl`, meaning calls to `setBaseUrl` (from Settings) do not update the existing `api` instance.
**Impact:** Setting a custom server URL in the UI will not work as expected unless the app is restarted or the instance is cleared.
**File:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`

### 3. Missing DSL Implementation
**Description:** The "DSL Engine" mentioned as a core feature in `package.json` consists of empty directories.
**Impact:** Core value proposition is currently non-existent.
**File:** `mr_notifier-server/src/dsl/*`

## Minor Issues

### 1. Hardcoded Password Validation
**Description:** Password validation logic is duplicated and hardcoded in the route handler instead of using the existing Zod schema or a utility.
**File:** `mr_notifier-server/src/modules/auth/auth.routes.ts`

### 2. Version Inconsistency
**Description:** `do-this.md` claims version 3.1.0, but `package.json` and `index.ts` report 3.0.0.
**File:** `mr_notifier-server/package.json`, `mr_notifier-server/src/index.ts`

### 3. Sync Logic Missing Pagination
**Description:** The sync endpoint returns all notifications at once. As the database grows, this will lead to performance degradation and memory issues on the mobile client.

## Long-term Risks
- **Scalability:** The current sync model (full fetch/merge) is not sustainable for high-frequency notification streams.
- **Maintainability:** The project is split between two Android folders (`mr_notifier` and `forked-version`), leading to confusion about which is the source of truth.
- **Security Debt:** Building a "no-login" mode as a primary feature makes it difficult to retrofit proper security later without breaking the user experience.

## Suggested Roadmap Updates
1. **Immediate:** Apply JWT middleware to all `/api/*` routes.
2. **Immediate:** Fix `sync_meta` logic to read from the database, not `mockSyncState`.
3. **Short-term:** Fix `ApiClient` lazy initialization bug in Android.
4. **Short-term:** Migrate from in-memory storage to Room (or a non-kapt alternative like SQLDelight if Termux support is critical).
5. **Mid-term:** Implement the DSL Engine to fulfill the project's purpose.

## Score: 45/100
*The project is functionally impressive for a Termux-developed app but is currently a "Security Nightmare" in its present state due to the intentional bypass of authentication and significant logic flaws in the production sync path.*
