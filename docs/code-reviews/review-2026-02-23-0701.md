# Code Audit Report: SignalNest (mr_notifier)

**Date:** 2026-02-23  
**Auditor:** Gemini CLI  
**Scope:** Full Repository Audit (mr_notifier-server, mr_notifier, forked-version)

---

## Executive Summary

SignalNest is a notification aggregation and delivery system. The project is currently in a transitional state, undergoing a major refactoring to move from a "no-login" prototype to an industry-standard production application. The backend is built with Hono (TypeScript/Node.js) and the Android app uses Jetpack Compose (Kotlin).

While the architectural blueprint (as seen in `REFACTORING_PLAN.md`) is solid, the implementation is currently highly experimental. The backend suffers from critical runtime errors in core routes and a brittle abstraction between its mock and production (Supabase) data layers. The Android application in `forked-version` is much more advanced than the one in the root `mr_notifier` directory, but it still lacks implementation for most of its promised features (DSL, Todos, Sources).

---

## Critical Issues

### 1. Runtime Reference Error in Notifications API
In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the `GET /` route attempts to use variables `start` and `end` which are never declared.
```typescript
// mr_notifier-server/src/modules/notifications/notifications.routes.ts
let query = supabase
  .from('notifications')
  .select('*', { count: 'exact' })
  .order('created_at', { ascending: false })
  .range(start, end); // start and end are undefined
```
**Impact:** The notifications list endpoint will crash in production mode.

### 2. State Inconsistency in Sync Versioning
In `mr_notifier-server/src/modules/sync/sync.routes.ts`, the `currentVersion` is calculated using an in-memory `mockSyncState` map, even when the server is in Supabase (production) mode.
```typescript
// mr_notifier-server/src/modules/sync/sync.routes.ts
const currentVersion = (mockSyncState.get(user_id)?.currentVersion || 0) + 1;
await supabase
  .from('sync_meta')
  .upsert({ ... });
```
**Impact:** On every server restart, the sync version resets to 1. This will cause clients to receive duplicate or inconsistent data during synchronization.

---

## Major Issues

### 1. Massive Code Duplication (Mock vs. Real)
Every route in the backend replicates its logic for `isMock()` mode. This violates the DRY (Don't Repeat Yourself) principle and makes the code extremely difficult to maintain. A repository or service layer should abstract the data source.

### 2. "Ghost" Implementation
Many directories in `mr_notifier-server/src/dsl` and `mr_notifier-server/src/modules` are empty or contain only placeholders. The project has the "structure" of a complex system (Lexer, Parser, Compiler, Validator) without any of the actual logic.

### 3. Data Loss in Development
As noted in the project's own `REFACTORING_PLAN.md`, the reliance on in-memory storage for the mock backend means all notifications and user data are lost on restart. This hinders effective frontend development.

---

## Minor Issues

- **Hardcoded User IDs:** Several API endpoints default `user_id` to `0` or `parseInt` it without proper validation or session binding.
- **Incomplete UI Screens:** The Android app's navigation drawer includes links to "Sources", "Schemas", and "Logs", all of which show static placeholder screens.
- **Termux Constraints:** The project avoids `Room` and `kapt` due to Termux compiler limitations, which is understandable for the environment but adds technical debt for standard Android deployment.

---

## Long-term Risks

- **Scalability:** The current sync logic (fetching all notifications since a timestamp) will become a performance bottleneck as the `notifications` table grows.
- **Security:** The `auth/me` route verifies JWTs but doesn't handle token revocation or robust error states for expired tokens.
- **Platform Divergence:** Having `mr_notifier` and `forked-version` as separate top-level directories is confusing. The project should consolidate into a single source of truth.

---

## Suggested Roadmap Updates

1. **Immediate:** Fix the `ReferenceError` in `notifications.routes.ts` and the `currentVersion` logic in `sync.routes.ts`.
2. **Architecture:** Implement a Data Access Object (DAO) or Repository pattern in the backend to eliminate the if/else mock checks in routes.
3. **Database:** Switch from in-memory mock to a local SQLite file to persist data during development on Termux.
4. **Consolidation:** Move the `forked-version` code into the main `mr_notifier` app and delete the redundant skeleton.
