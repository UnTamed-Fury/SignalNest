# Code Audit Report: SignalNest (mr-notifier)
**Date:** 2026-02-23 02:30 UTC  
**Scope:** Full repository audit (Server, Android App, Refactored App)  
**Status:** In-Development / Refactoring Phase

---

## Executive Summary
The project is currently undergoing a significant architectural transition from a basic prototype (`mr_notifier`) to a production-grade system (`forked-version` / `SignalNest`). The server is built on Hono with a modular design but suffers from significant "mock vs. production" logic pollution and critical copy-paste bugs in its Supabase implementation. The Android application in `forked-version` demonstrates high architectural standards (MVVM, Repository pattern, WorkManager), but lacks the necessary test coverage and database persistence on the backend to be considered 'A-grade' production standard.

---

## Critical Issues

### 1. Server-Side Logic Bugs (Supabase Mode)
*   **Undefined Variables in Pagination:** In `notifications.routes.ts`, the Supabase `range(start, end)` call uses `start` and `end` variables that are only defined within the `isMock()` block, causing a crash in production mode.
*   **Broken Auth Logic:** In `auth.routes.ts`, the check `if (!isMock)` is a function reference check (always true) instead of a call `!isMock()`. This leads to Supabase update attempts in mock mode.
*   **Cross-Pollination of Mock State:** In `sync.routes.ts`, the "Real Supabase mode" incorrectly uses `mockSyncState` to calculate the `currentVersion`, meaning sync versions will be lost on server restart even when using a real database.

### 2. Missing Persistence (Android & Backend)
*   **Broken Backend Storage:** Per `REFACTORING_PLAN.md`, notifications are lost between API requests in the current backend implementation.
*   **Lack of Database on Android:** The project avoids Room/kapt due to Termux limitations, relying on in-memory storage for notifications, which is a high risk for data loss on the client side.

---

## Major Issues

### 1. Total Lack of Automated Testing
*   While `vitest` and `junit` are in the dependency lists, a codebase search revealed **zero** actual test files in the project source.
*   The DSL engine (lexer, parser, compiler) is particularly vulnerable without unit tests to verify complex logic.

### 2. Mock Logic Pollution
*   The `isMock()` branching is scattered throughout every route module. This should be abstracted into a Service or Repository layer to keep routes clean and prevent the bugs identified in the Critical section.

### 3. Non-Persistent Notifications (Legacy App)
*   The `mr_notifier` app uses `Handler.postDelayed` for scheduling, which is not persistent across app kills or reboots. (Partially addressed in `forked-version` via `WorkManager`).

---

## Minor Issues

### 1. Placeholder Bloat
*   Numerous directories in `src/dsl` and `src/modules` are empty placeholders, creating a misleading sense of project completion.
*   Naming inconsistencies: The project refers to itself as "mr-notifier", "Notifier 300", and "SignalNest".

### 2. Unoptimized DB Queries
*   The server frequently uses `select('*')` in Supabase queries. For production scaling, this should be restricted to specific columns.

---

## Technical Debt & Risks

*   **Termux Constraints:** The decision to avoid Room/kapt due to Termux environment constraints is a significant long-term risk. It forces the use of less robust storage mechanisms that may not scale to production requirements.
*   **DSL Maintainability:** Developing a custom DSL stack without tests or formal documentation is a major maintenance bottleneck.

---

## Suggested Roadmap Updates

1.  **URGENT:** Fix the Supabase pagination and auth bugs in the server.
2.  **Infrastructure:** Finalize the Supabase integration to replace in-memory storage.
3.  **Testing:** Implement core unit tests for the DSL lexer/parser and the Auth service.
4.  **Abstraction:** Refactor the server to use a Repository pattern, removing `isMock()` checks from the routes.
5.  **Persistence:** Investigate alternative local storage for Android (e.g., SQLDelight or manual SQLite) that works in Termux without kapt.
