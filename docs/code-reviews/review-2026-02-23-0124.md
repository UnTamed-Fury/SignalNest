# Code Audit Report: SignalNest / MR Notifier
**Date:** 2026-02-23  
**Auditor:** Gemini CLI  
**Scope:** Recursive Static Analysis of `mr_notifier` repository.

---

## Executive Summary
The project, referred to as "SignalNest" in code and "MR Notifier" in the repository, is currently in an early development phase with significant architectural gaps. While the Android application (`forked-version`) shows a functional UI structure and basic network integration, the backend server is largely a collection of empty scaffolds. Most critically, the project suffers from **severe security vulnerabilities** and **zero test coverage**, making it unfit for production use in its current state.

**Overall Score: 28/100**

---

## Critical Issues (Must Fix Immediately)

### 1. Missing Authentication Middleware (Security)
The server implementation (`mr_notifier-server`) lacks any JWT verification or authentication guards for the `/api/*` routes.
- **Impact:** Any user can access, modify, or create notifications for any other user by simply providing a `user_id` in the query or body.
- **File:** `mr_notifier-server/src/index.ts`, `notification.routes.ts`, `sync.routes.ts`.

### 2. Silent Mock Mode Fallback (Reliability)
The database layer automatically falls back to a volatile in-memory "mock mode" if environment variables (Supabase keys) are missing or invalid.
- **Impact:** Data loss on server restart. In production, a misconfigured environment variable would result in a "working" but data-loss-prone system without immediate warning.
- **File:** `mr_notifier-server/src/core/db/database.ts`.

### 3. Zero Test Coverage (Quality)
There are no unit, integration, or UI tests in either the Android apps or the backend server.
- **Impact:** High risk of regressions, no verification of business logic, and difficult maintenance.
- **Status:** `src/test` and `src/androidTest` directories are completely missing in Android; server `test` script exists but points to no files.

---

## Major Issues

### 1. Massive Empty Scaffolding (Architecture)
The backend project features a complex directory structure for a "DSL Engine" and several modules (`canonical`, `delivery`, `raw_events`, `sources`, `dsl`) that contain **zero implementation**.
- **Impact:** misleading architectural complexity; the "DSL Engine" mentioned in documentation and `package.json` does not exist.
- **Directory:** `mr_notifier-server/src/dsl/`, `mr_notifier-server/src/modules/`.

### 2. Tight Coupling and Code Duplication
Business logic is implemented directly in Hono routes with parallel code paths for "Mock Mode" and "Supabase Mode".
- **Impact:** Hard to maintain and extend. Violates DRY (Don't Repeat Yourself) and SRP (Single Responsibility Principle).
- **File:** `auth.routes.ts`, `notifications.routes.ts`.

### 3. Naming Inconsistency
The project is referred to by three different names: `SignalNest`, `MR Notifier`, and `Fury`. 
- **Impact:** Confusion in documentation, package names (`com.signalnest.app` vs `fury.notifier`), and API endpoints.

---

## Minor Issues / Technical Debt

- **Unrestricted CORS:** `Access-Control-Allow-Origin: *` is used globally.
- **In-Memory Android Storage:** The Android app relies on in-memory storage for notifications because "Room/kapt doesn't work on Termux."
- **Missing Documentation:** No internal developer guides, API specifications (only placeholders), or deployment instructions.
- **Outdated Project Root:** The `mr_notifier/` (Android) folder seems to be a legacy version compared to `forked-version/`.

---

## Long-term Risks

1. **Scalability:** The current sync logic (timestamp-based) will struggle with high volume and edge cases (clock skew, deleted items).
2. **Platform Constraints:** Development on Termux is severely limiting the tech stack (no Room, no kapt), which will lead to more technical debt as the app grows.
3. **DSL Maintenance:** Implementing a custom DSL from scratch is a high-maintenance effort. Without an existing implementation, the feasibility of this feature is unproven.

---

## Suggested Roadmap Updates

1. **Security First:** Implement a global JWT authentication middleware for all `/api` routes in the backend.
2. **Database Abstraction:** Refactor the backend to use a proper Repository pattern that abstracts away the difference between Supabase and Mock modes.
3. **Verification:** Add at least basic unit tests for the Auth and Sync logic.
4. **Cleanup:** Delete empty folders and unused scaffolds to reduce cognitive load.
5. **Unified Naming:** Decide on one project name and refactor package names and logs to match.
6. **KSP Migration:** Investigate using Room with KSP (Kotlin Symbol Processing) to see if it bypasses Termux's kapt limitations.
