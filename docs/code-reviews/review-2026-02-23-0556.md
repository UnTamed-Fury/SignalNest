# Code Audit Report - SignalNest / MR Notifier
**Date:** 2026-02-23 05:56:33 UTC
**Auditor:** Gemini CLI Audit Bot

---

## Executive Summary
The SignalNest (formerly MR Notifier) project is in an early MVP stage. While a basic end-to-end flow is theoretically possible, the current codebase is extremely sparse, lacks critical implementations, and contains production-breaking bugs. The project has zero automated test coverage, and the server architecture relies heavily on fragile conditional branching for "mocking." Significant refactoring and implementation work are required to reach a production-grade standard.

---

## Critical Issues (Immediate Fix Required)
1. **Server-Side Reference Errors:** In `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, the `GET /` route uses `start` and `end` variables in the Supabase query which are never defined. This will cause the server to crash or return a 500 error in any environment where `isMock()` is false (production/staging).
2. **Lack of Persistence (Android):** The Android application (`forked-version`) currently uses in-memory `MutableStateFlow` in the `SignalNestRepository` for all data storage. All notifications, notes, and todos are lost every time the app process is killed.
3. **Empty Core Modules:** Numerous core directories (`adapters`, `dsl`, `modules/canonical`, `modules/delivery`, etc.) are completely empty. The "DSL Engine" mentioned in the `package.json` description does not exist in the source code.

---

## Major Issues
1. **Zero Test Coverage:** No unit, integration, or end-to-end tests exist for either the server or the Android application.
2. **Inefficient API Usage:** The Android `HomeViewModel` iterates over notifications and calls `markAsRead` individually instead of using the repository's (and server's) available `markAllAsRead` bulk endpoint.
3. **Fragile Mocking Logic:** The server uses `if (isMock())` branches inside every route handler. This couples testing/development logic with production code and significantly increases the chance of environment-specific bugs.
4. **Missing Device Registration:** The Android app references an `api/devices/register` endpoint which is not implemented in the server's `index.ts` or routes.

---

## Minor Issues / Technical Debt
1. **Inconsistent Data Stores:** Both `database.ts` and `auth.routes.ts` maintain separate, un-unified mock data stores.
2. **Open CORS Policy:** The server has `Access-Control-Allow-Origin: *` which should be restricted to known clients in a production environment.
3. **Security Gaps:** While JWT is used, there is no centralized middleware to protect `/api` routes; some routes appear unprotected or rely on manual checks.
4. **Naming Inconsistency:** The project is inconsistently referred to as `mr-notifier`, `mr_notifier`, and `signalnest`.

---

## Long-term Risks
- **Scalability:** The current reliance on in-memory structures (even for mocks) will lead to performance degradation as data grows.
- **Maintainability:** Without tests and with many "placeholder" routes, the codebase will become increasingly difficult to evolve without introducing regressions.
- **Security:** The lack of robust input validation (beyond basic Zod usage in some places) and open CORS policies pose risks for public deployment.

---

## Suggested Roadmap Updates
1. **PRIORITY:** Calculate `start` and `end` for pagination in `notifications.routes.ts`.
2. **PRIORITY:** Implement Room database for persistence in the Android application.
3. **REFACTOR:** Move mocking logic into a separate repository/adapter layer to clean up route handlers.
4. **TESTING:** Implement at least 50% unit test coverage for core business logic (Auth, Sync, Notifications).
5. **IMPLEMENT:** Fill the `adapters` and `dsl` modules to meet the project's stated goals.

---
