# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23 04:37 UTC  
**Scope:** Full Repository (Server + Android App)  
**Auditor:** Gemini CLI

---

## 1. Executive Summary
SignalNest is an ambitious event intelligence and notification system. The project exhibits an exceptionally strong architectural foundation, particularly in its database schema and Android application structure. However, the current state of the implementation is largely incomplete, with significant gaps between the planned "A-grade" production standard and the actual code. The most critical findings are a complete lack of automated tests, the use of volatile in-memory storage for persistent data, and overly permissive security configurations.

---

## 2. Critical Issues (Must Fix Immediately)
| Issue | Description | Impact |
|-------|-------------|--------|
| **Zero Test Coverage** | No unit, integration, or UI tests found in `mr_notifier-server` or `forked-version`. | Extreme risk of regressions and high maintenance cost. |
| **In-Memory Storage** | Both the server (in mock mode) and the Android app (SignalNest version) rely on in-memory storage for notifications, notes, and todos. | Total data loss on app restart or server reboot. Prevents production use. |
| **Insecure CORS Policy** | `mr_notifier-server/src/index.ts` uses `Access-Control-Allow-Origin: '*'`. | Opens the server to Cross-Origin Resource Sharing (CORS) attacks. |

---

## 3. Major Issues
| Issue | Description | Impact |
|-------|-------------|--------|
| **Incomplete Implementation** | Many directories in the server (`dsl/`, `adapters/`) are empty placeholders. The DSL engine—a core feature—is entirely missing. | The project is currently a skeleton rather than a functional product. |
| **Mock Mode Disparity** | The server's `createMockSupabase` is primitive and does not reflect the complexity of the actual `schema.sql`. | Development in mock mode will lead to bugs that only surface in production. |
| **Volatile Notification Scheduling** | The "test" app (`mr_notifier/`) uses `Handler.postDelayed` for scheduling, which fails if the process is killed. | Unreliable notification delivery for the primary use case. |

---

## 4. Minor Issues & Technical Debt
| Issue | Description | Impact |
|-------|-------------|--------|
| **Confusing Project Structure** | Coexistence of `mr_notifier/` (test app) and `forked-version/` (real app) is confusing for new contributors. | Increased onboarding time and potential for misdirected effort. |
| **Hardcoded salt rounds** | `bcrypt.hash` uses a hardcoded factor of 10. | Minor security/performance tradeoff; should be configurable. |
| **Regex for Passwords** | The special character regex in `auth.routes.ts` is overly restrictive. | Potential UX issue for users with complex passwords. |

---

## 5. Architectural Evaluation
### Backend (SignalNest Server)
- **Strengths:** Modern stack (Hono + Supabase + Zod). Excellent schema design supporting event-driven processing and versioned sync.
- **Weaknesses:** Gap between schema complexity and actual route implementation is massive.

### Frontend (SignalNest Android)
- **Strengths:** Follows industry standards (Clean Architecture, MVI/MVVM pattern, Jetpack Compose, WorkManager). The `syncNotifications` logic is well-designed.
- **Weaknesses:** Lack of local persistence (Room) makes it non-viable as a notification client.

---

## 6. Long-term Risks
- **Scalability of DSL:** Without the DSL engine implemented, it's impossible to evaluate the performance impact of custom rules.
- **Maintenance Burden:** The high level of architectural complexity without supporting tests will lead to a "fragile" codebase as features are added.

---

## 7. Suggested Roadmap Updates
1. **Implementation of Room Persistence:** Replace in-memory storage in the Android app immediately to enable offline use.
2. **Database Integration:** Prioritize connecting the backend to a real Supabase/Postgres instance and removing mock logic from routes.
3. **Test-Driven Development:** Establish a test suite for the Auth module and Sync logic before proceeding with new features.
4. **DSL Engine Prototype:** Implement a basic version of the DSL engine to validate the architectural design.
5. **Security Hardening:** Restrict CORS to specific domains and move the Supabase service key to a more secure environment (e.g., Vault).

---
