## Executive Summary
The project is currently in a "stunt-double" phase—it looks like a complete system from a directory and schema perspective, but the actual implementation is missing core functionality (DSL engine, adapters) and contains critical logic flaws. The Android app's notification scheduling is fundamentally unreliable, and the server's production path (Supabase) contains reference errors and state-management bugs. Significant effort is required to move this from a prototype to an 'A-grade production standard'.

---

## Critical Issues

### 1. Unreliable Notification Scheduling (Android)
**File:** `fury/notifier/notification/AppNotificationManager.kt`  
**Description:** Notifications are scheduled using `android.os.Handler(context.mainLooper).postDelayed`.  
**Risk:** In `HomeScreen.kt`, the app calls `activity?.finishAndRemoveTask()` immediately after scheduling. On many Android versions/devices, finishing the last activity allows the system to kill the process. If the process is killed, the `Handler` is destroyed, and the notification will never fire.  
**Recommendation:** Replace `Handler` with `AlarmManager` for precise timing or `WorkManager` for guaranteed execution.

### 2. Reference Errors in Production Path (Server)
**File:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`  
**Description:** The variables `start` and `end` used in the Supabase query are defined using `const` inside a different scope (`if (isMock())`).  
**Risk:** The API will crash with a `ReferenceError` when running in Supabase mode (production).  
**Recommendation:** Move variable definitions to the common scope of the request handler.

### 3. State-Persistence Bug in Sync Logic (Server)
**File:** `mr_notifier-server/src/modules/sync/sync.routes.ts`  
**Description:** The sync logic updates Supabase's `sync_meta` using `mockSyncState.get(user_id)?.currentVersion`.  
**Risk:** `mockSyncState` is an in-memory Map. If the server restarts, the version counter resets to 0, but the data in Supabase remains. This will cause versioning conflicts and incorrect synchronization for clients.  
**Recommendation:** Query the current version from the database before incrementing and saving.

## Major Issues

### 1. Missing Core Engine (DSL)
**Directory:** `mr_notifier-server/src/dsl/`  
**Description:** All DSL-related directories (lexer, parser, compiler, etc.) are empty.  
**Risk:** The project's main value proposition ("Event Intelligence System with DSL Engine") is currently non-existent in the codebase.  
**Recommendation:** Prioritize the implementation of the Lexer and Parser.

### 2. Zero Automated Testing
**Description:** No tests found (`**/*.test.ts` or `**/*Test.kt`).  
**Risk:** High regression risk, especially given the dual-mode (Mock/Supabase) logic which doubles the surface area for bugs.  
**Recommendation:** Implement Vitest for the server and Compose UI tests for the app.

### 3. Missing Network Permissions (Android)
**File:** `mr_notifier/app/src/main/AndroidManifest.xml`  
**Description:** `android.permission.INTERNET` is missing.  
**Risk:** The app cannot communicate with the SignalNest server.  
**Recommendation:** Add the internet permission to the manifest.

---

## Minor Issues

### 1. Hardcoded Pagination Defaults
**File:** `notifications.routes.ts`  
**Description:** `page_size` defaults to 50 without a maximum cap.  
**Risk:** Potential for DoS or performance degradation if a client requests a massive page size.

### 2. Wide CORS Policy
**File:** `index.ts`  
**Description:** `Access-Control-Allow-Origin` is set to `*`.  
**Risk:** While acceptable for a public API, it should be restricted to known domains in production to mitigate CSRF-like risks.

### 3. Bcrypt Salt Rounds
**File:** `auth.routes.ts`  
**Description:** Uses 10 rounds of bcrypt.  
**Risk:** 10 rounds is the bare minimum today; 12 or higher is recommended for future-proofing against faster hardware.

---

## Long-term Risks
- **Mock Mode Drift:** The system maintains two parallel implementations for almost every route (Mock vs. Supabase). As the system grows, keeping these in sync will become a major maintenance burden.
- **Statelessness Violation:** The server relies on in-memory maps (`mockUsers`, `mockNotifications`) for its "Mock Mode", making it impossible to scale horizontally or survive restarts without data loss.

---

## Suggested Roadmap Updates
1. **Immediate:** Fix the `ReferenceError` in `notifications.routes.ts` and the `AlarmManager` implementation in the Android app.
2. **Short-term:** Implement `INTERNET` permission and basic Retrofit/Ktor client in Android to talk to the server.
3. **Mid-term:** Build the MVP of the DSL Lexer and Parser to validate the "Event Intelligence" concept.
4. **Mid-term:** Introduce a `Repository` pattern to abstract the data source, eliminating the `if (isMock())` blocks inside route handlers.
