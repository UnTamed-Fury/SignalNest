# Code Audit Report: MR Notifier / SignalNest
**Date:** 2026-02-23
**Auditor:** Gemini CLI Agent
**Score:** 45/100

---

## 1. Executive Summary
The project is currently transitioning from a simple notification scheduler (MR Notifier) to a more robust event intelligence platform (SignalNest). The `forked-version` contains the modern Jetpack Compose mobile application, while the `mr_notifier-server` provides a Hono-based TypeScript backend. 

While the technology stack is modern (Zod, Hono, Supabase, Jetpack Compose, Kotlin Coroutines), the project is currently in a "pre-alpha" state regarding reliability. The total absence of automated testing and the incomplete state of core features like the DSL engine and the synchronization protocol are significant hurdles to production readiness.

---

## 2. Critical Issues

### 2.1 Zero Test Coverage
*   **Description:** No unit tests, integration tests, or UI tests were found in either the backend or frontend codebases.
*   **Impact:** Any change has a high risk of introducing regressions. Technical debt will accumulate rapidly, and refactoring (which is needed) will be extremely dangerous.
*   **Recommendation:** Immediately implement a testing strategy. Start with unit tests for the server's auth logic and the mobile app's repository layer.

### 2.2 Incomplete Sync Protocol
*   **Description:** The `/api/sync` endpoint only fetches notifications where `created_at > last_sync_timestamp`. 
*   **Impact:** 
    *   **Updates are lost:** If a notification is marked as read on the web/another device, the mobile app will not receive this update during sync.
    *   **Deletions are lost:** The server does not communicate deleted notification IDs; they remain on the mobile device indefinitely.
*   **Recommendation:** Redesign the sync protocol to use a `sync_version` (sequence number) or a `deleted_at` timestamp. Return a list of `updated` and `deleted` IDs.

---

## 3. Major Issues

### 3.1 Silent Mock Mode Fallback
*   **Description:** The server automatically switches to an in-memory "mock mode" if Supabase environment variables are missing or incorrectly configured.
*   **Impact:** In a production environment, a misconfiguration (e.g., a typo in `SUPABASE_SERVICE_ROLE_KEY`) will result in the server starting and accepting data that is never persisted, leading to total data loss on restart.
*   **Recommendation:** Explicitly require a `DATABASE_MODE=mock|production` environment variable and fail fast if the required credentials for the chosen mode are missing.

### 3.2 Duplicated Business Logic (Mock vs. Production)
*   **Description:** Every route (Auth, Notifications, Sync) contains `if (isMock()) { ... } else { ... }` blocks.
*   **Impact:** High maintenance overhead. Adding a feature requires implementing it twice. It increases the chance of the mock implementation diverging from production behavior, making "local development" misleading.
*   **Recommendation:** Implement a Repository Pattern. Define an interface for data access and have `SupabaseRepository` and `InMemoryRepository` implementations.

### 3.3 Empty DSL Engine Architecture
*   **Description:** The directory structure for a DSL compiler, parser, and executor exists but contains no implementation files.
*   **Impact:** The "SignalNest Rule Language (SNRL)" is a core value proposition of the project but is currently non-existent.
*   **Recommendation:** Prioritize the DSL implementation or remove the empty directories to reduce architectural "noise."

### 3.4 Inconsistent Branding and Naming
*   **Description:** The codebase refers to the project as "MR Notifier," "Notifier 300," and "SignalNest" interchangeably.
*   **Impact:** Confusing for developers and users. High risk of shipping mismatched strings in the UI.
*   **Recommendation:** Execute a global rename to "SignalNest" (as indicated by the newer `forked-version`).

---

## 4. Minor Issues

### 4.1 Manual Validation in Auth
*   **Description:** `auth.routes.ts` uses manual regex and length checks for email and password validation.
*   **Impact:** Inconsistent with the Zod-based validation used for environment variables.
*   **Recommendation:** Use Zod schemas for all request body validation.

### 4.2 Lack of Mock Persistence
*   **Description:** Mock data is lost on every server restart.
*   **Impact:** Hinders development of features that require persistent state (like the sync flow).
*   **Recommendation:** Allow the mock mode to save/load from a local JSON file if configured.

### 4.3 Mobile UI Placeholders
*   **Description:** Multiple core screens (Sources, Schemas, Logs) are just placeholders.
*   **Impact:** The app feels incomplete and non-functional for its primary intended use case (event intelligence).

---

## 5. Long-term Risks
*   **Scalability of Mock Logic:** As the data model grows, maintaining a manual in-memory mock that mirrors Supabase behavior will become unsustainable.
*   **Vendor Lock-in:** The heavy reliance on Supabase-specific features (like their `from().select()` syntax) makes it difficult to move to a different provider if needed.
*   **Technical Debt in Sync:** Fixing the sync protocol later will require breaking changes to the mobile app's local database schema.

---

## 6. Suggested Roadmap Updates
1.  **Test Infrastructure (Week 1):** Setup Vitest for the server and JUnit/Espresso for the Android app.
2.  **Repository Refactor (Week 2):** Abstract database logic to remove `if(isMock)` from routes.
3.  **Sync Protocol v2 (Week 3):** Implement sequence-based synchronization with support for updates and deletions.
4.  **DSL Foundations (Week 4):** Implement the Lexer and Parser for the SNRL.
