# Code Audit Report: SignalNest (mr_notifier)

**Date:** 2026-02-23  
**Auditor:** Gemini CLI  
**Scope:** Full repository analysis (Backend & Android App)  
**Score:** 45/100

---

## Executive Summary
SignalNest is a personal notification aggregation system in its early prototype phase. While the project shows a clear architectural vision (Node.js/Hono backend and Kotlin/Compose Android app), it currently suffers from critical architectural compromises and missing core features. The transition from the legacy `mr_notifier` to the `forked-version` (SignalNest) is incomplete. Major blockers include the lack of persistent storage and a total absence of automated testing.

---

## Critical Issues

### 1. Lack of Persistent Storage (Architectural Weakness)
- **Backend:** The server heavily relies on "mock mode" with in-memory storage (`mock-store.ts`). Even when using Supabase, several operations fallback to in-memory maps, causing data loss on server restarts.
- **Android App:** `SignalNestRepository` uses `MutableStateFlow` for notifications, notes, and todos. All data is lost when the app process is killed. The project documentation cites Termux limitations (`kapt` incompatibility) for not using Room, which is a major technical hurdle for production viability.

### 2. Missing Core Modules (Incomplete Implementation)
- The `dsl` directory (lexer, parser, compiler) is entirely empty, yet referenced as a core feature for notification processing.
- Adapters for GitHub, Discord, and Webhooks are empty shells, meaning no external events can currently be ingested.

### 3. Total Absence of Automated Tests (Quality Risk)
- No unit, integration, or UI tests were found in the entire repository.
- There is no CI/CD pipeline configured to run tests (only build workflows).

---

## Major Issues

### 1. Security Risks
- **Insecure CORS:** `mr_notifier-server/src/index.ts` uses `Access-Control-Allow-Origin: *`, which is overly permissive.
- **JWT Secret on Client:** `ApiClient.kt` in the Android app contains logic to save and load a `jwt_secret`. The client should never handle the signing secret; it should only store the issued token.
- **isMock Bug:** In `auth.routes.ts`, `isMock` (a function) is checked as a boolean (`if (!isMock)`), which will always evaluate to true, potentially causing logic errors in production environments.

### 2. FCM Synchronization Gap
- `SignalNestMessagingService` displays system notifications but does not update the local repository. Users must manually sync to see new notifications in the app UI, leading to a disconnected user experience.

### 3. Hardcoded Configurations
- Multiple instances of hardcoded `userId = 0` and localhost IP addresses (`10.0.2.2`) exist in the production source code.

---

## Minor Issues

- **Technical Debt:** The codebase is littered with TODOs regarding device registration and Room integration.
- **Refactoring State:** The repository structure is fragmented between `mr_notifier` (legacy) and `forked-version` (active), causing confusion.
- **Validation Logic:** Password validation is implemented manually in both the backend and frontend; using a standard library (like Zod for backend) would be more robust.

---

## Long-term Risks

- **Platform Lock-in:** The decision to compromise architecture (avoiding Room/kapt) due to Termux environment limitations will severely hinder the app's scalability and reliability on standard Android devices.
- **Maintainability:** Without tests and with a custom (currently empty) DSL, the project will become increasingly difficult to maintain as complexity grows.

---

## Suggested Roadmap Updates

1. **Implement Database Persistence:** Move away from in-memory storage. If Room is not possible on Termux, consider using a raw SQLite wrapper or a remote-only mode with aggressive caching.
2. **Fix Security Flaws:** Remove `jwt_secret` from the client and tighten CORS settings.
3. **Establish Testing Suite:** Implement at least 50% unit test coverage for ViewModels and Backend routes.
4. **Complete DSL/Adapters:** Prioritize the implementation of the GitHub adapter to make the system functional for event ingestion.
5. **Consolidate Repository:** Remove the legacy `mr_notifier` folder and rename `forked-version` to `app`.
