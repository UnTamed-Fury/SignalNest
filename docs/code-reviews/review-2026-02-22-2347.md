# Code Review Report - 2026-02-22-2347

## Executive Summary
The project, recently rebranded as **SignalNest** (formerly `mr_notifier`), is in a transitional state from a prototype to a production-ready application. While it employs modern frameworks (Hono for backend, Jetpack Compose for Android), it currently suffers from critical security vulnerabilities and architectural gaps that prevent it from meeting 'A-grade production standards'. The most pressing issues are the lack of authentication on core API routes and the absence of automated testing.

**Overall Score: 45/100**

---

## Critical Issues

### 1. Unprotected API Routes
**Location:** `mr_notifier-server/src/index.ts`, `notification.routes.ts`, `sync.routes.ts`
**Description:** Core API routes for notifications and synchronization do not have authentication middleware applied. Although an authentication module exists, it is only used for `/auth/me`. Any user can access, list, and create notifications if they know the endpoint.
**Risk:** Data breach, unauthorized access to user notifications, and potential spam/DOS of the notification system.

### 2. Client-Side JWT Secret Storage
**Location:** `forked-version/app/src/main/java/com/signalnest/app/data/api/ApiClient.kt`
**Description:** The Android application stores a `JWT_SECRET` in its `DataStore`. In a standard secure architecture, the JWT secret should be known only to the server. If the client knows the secret, a malicious user could reverse-engineer the app and forge valid JWT tokens for any user ID.
**Risk:** Total compromise of the authentication system.

### 3. Zero Automated Testing
**Location:** Entire Repository
**Description:** There are no unit, integration, or UI tests in either the backend or the frontend. The project relies entirely on manual verification.
**Risk:** High regression rate, brittle codebase, and difficulty in scaling development.

---

## Major Issues

### 1. Production Bug in Sync Versioning
**Location:** `mr_notifier-server/src/modules/sync/sync.routes.ts`
**Description:** In the "Real Supabase mode" branch of the sync endpoint, the `currentVersion` is calculated using `mockSyncState.get(user_id)`. Since `mockSyncState` is an in-memory map, it resets on every server restart and is not shared across server instances.
**Risk:** Desynchronization between client and server, potentially leading to missed notifications or duplicate data.

### 2. In-Memory Storage in Android App
**Location:** `forked-version/app/src/main/java/com/signalnest/app/data/repository/SignalNestRepository.kt`
**Description:** The primary repository for the Android app uses an in-memory `MutableStateFlow` to store notifications. All data is lost when the app process is terminated by the OS or the user.
**Risk:** Severe data loss; users will lose all notification history and local notes/todos upon app restart.

### 3. DSL Feature is a Skeleton
**Location:** `mr_notifier-server/src/dsl/`
**Description:** The DSL engine (lexer, parser, compiler, etc.) consists entirely of empty directories. The project's marketing as an "Event Intelligence System with DSL Engine" is currently inaccurate as the core logic is missing.
**Risk:** Misleading documentation and incomplete feature set.

---

## Minor Issues

### 1. Inefficient Logger File Writing
**Location:** `mr_notifier-server/src/core/logger/logger.ts`
**Description:** The logger performs a dynamic `import('fs/promises')` inside the `writeToLogFile` function, which is called for every log entry if file logging is enabled.
**Risk:** Performance overhead during high-volume logging.

### 2. Brittle Mock/Real Logic Mix
**Location:** Server Routes
**Description:** The routes are heavily cluttered with `if (isMock())` blocks. This makes the code harder to read and increases the chance of bugs when transitioning between development and production modes.
**Recommendation:** Use the Strategy pattern or dependency injection to separate mock and real database implementations.

---

## Long-Term Risks
*   **Scalability**: The current sync logic (fetching all notifications since a timestamp without robust versioning) will become slow as user data grows.
*   **Maintainability**: Without tests, the complex DSL engine (once implemented) will be extremely difficult to maintain and debug.
*   **Branding Inconsistency**: The codebase still has references to `fury`, `mr_notifier`, and `SignalNest`. This needs to be unified.

---

## Suggested Roadmap Updates
1.  **Immediate**: Implement a global `authMiddleware` in the Hono server and apply it to all `/api/*` routes.
2.  **Immediate**: Remove `JWT_SECRET` from the Android app; the server should be the sole authority for token signing.
3.  **High Priority**: Replace in-memory storage in the Android app with a Room database.
4.  **High Priority**: Set up a CI/CD pipeline and start adding unit tests for the backend's core logic.
5.  **Medium Priority**: Implement the DSL engine to fulfill the project's primary value proposition.
6.  **Medium Priority**: Refactor the backend to use a clean repository pattern, abstracting the database layer to remove `if (isMock())` from route handlers.
