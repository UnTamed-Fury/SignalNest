# Code Audit Report: SignalNest (MR Notifier)
**Date:** 2026-02-23 02:43 UTC  
**Scope:** Full Repository (Server + Mobile App)  
**Status:** Scheduled 10-Minute Static Audit  

---

## 1. Executive Summary
SignalNest is in a critical transition phase, moving from a prototype to an industry-standard architecture. While the vision (clean architecture, Repository pattern, DSL engine) is sound, the current implementation is **not production-ready**. Major security vulnerabilities exist in the API layer, and technical debt is high due to platform-specific constraints (Termux). The project lacks any automated testing, which is the most significant risk to long-term stability.

---

## 2. Critical Issues (Action Required Immediately)

### 🚨 [C01] Broken Authorization Model
**Location:** `mr_notifier-server/src/modules/notifications/notifications.routes.ts`, `sync.routes.ts`  
**Impact:** Massive Data Leakage.  
**Description:** API endpoints for listing notifications and syncing data take a `user_id` as a query or body parameter and return data for that ID WITHOUT verifying if the authenticated user owns that ID. Furthermore, global authentication middleware is missing for the `/api` route prefix.  
**Recommendation:** Implement a global auth middleware that extracts `userId` from the JWT and ensures all queries are scoped to that ID. Remove `user_id` as a client-provided parameter where possible.

### 🚨 [C02] Missing Automated Tests
**Location:** Entire Repository.  
**Impact:** Regressions and Unreliable Deployments.  
**Description:** Zero unit, integration, or UI tests were found despite `package.json` referencing `vitest`. A project of this complexity (DSL, Sync logic) cannot be maintained safely without a test suite.  
**Recommendation:** Immediately implement the "Phase 4: Testing Strategy" from the `REFACTORING_PLAN.md`.

### 🚨 [C03] Hardcoded Secrets in Environment Templates
**Location:** `mr_notifier-server/.env`  
**Impact:** Security Compromise.  
**Description:** A `JWT_SECRET` is hardcoded in the `.env` file and potentially committed (though `.gitignore` exists, the template should be used).  
**Recommendation:** Ensure `.env` is never committed and use strong, randomly generated secrets in production.

## 3. Major Issues

### 🔴 [M01] Ghost Feature: DSL Engine
**Location:** `mr_notifier-server/src/dsl/`  
**Impact:** Misleading Documentation/Technical Debt.  
**Description:** The project claims to have a "DSL Engine" in `package.json` and directory structures, but the implementation directories are currently empty.  
**Recommendation:** If the feature is not yet implemented, mark it as "experimental/upcoming" in documentation to avoid confusion.

### 🔴 [M02] Volatile Storage (Termux Constraint)
**Location:** `forked-version/app/`  
**Impact:** Data Loss.  
**Description:** Due to kapt/Room issues on Termux, the app uses in-memory storage for notifications, notes, and todos. Data is lost on app process death.  
**Recommendation:** Explore `SqliteDatabase` (standard Android API) or `DataStore` (Protobuf/Preferences) as alternatives that don't require kapt to provide persistence on Termux.

### 🔴 [M03] Version Inconsistency
**Location:** `SignalNestApi.kt` (v3.1.0) vs `package.json` (v3.0.0).  
**Impact:** Integration Failures.  
**Description:** The mobile app expects a server version that doesn't match the actual server version.  
**Recommendation:** Synchronize versioning across the stack.

---

## 4. Minor Issues & Technical Debt

### 🟡 [L01] Naive Sync Logic
**Location:** `sync.routes.ts`  
**Description:** Relies solely on `created_at` timestamps for sync.  
**Risk:** Potential for missed events if multiple events share a timestamp or if database clock skew occurs. Use sequential `sync_version` or IDs.

### 🟡 [L02] Permissive CORS
**Location:** `mr_notifier-server/src/index.ts`  
**Description:** `Access-Control-Allow-Origin: '*'` is used.  
**Risk:** CSRF/Cross-origin risks if the server is used in a web context. Restrict to known origins.

### 🟡 [L03] Partial Result Wrapper Implementation
**Location:** `SignalNestRepository.kt`  
**Description:** Some methods use `Result<T>`, others don't.  
**Risk:** Inconsistent error handling in ViewModels.

---

## 5. Long-term Risks
- **Scalability:** The in-memory mock mode in the server will fail under load or with many concurrent users.
- **Maintainability:** Without Room, the data layer in the Android app will become extremely complex to manage manually (SQL strings, cursor handling).
- **Security:** The lack of strict input validation beyond Zod (e.g., sanitizing notification bodies) could lead to XSS if notifications are ever rendered in a web view.

---

## 6. Suggested Roadmap Updates
1. **Priority 1:** Secure the API (Auth middleware + Ownership checks).
2. **Priority 2:** Implement persistence for the Android app (SQLite/DataStore).
3. **Priority 3:** Establish a baseline test suite (at least for `SignalNestRepository` and `AuthRoutes`).
4. **Priority 4:** Implement the DSL compiler and executor to fulfill the project's core value proposition.
