# Code Audit Report: SignalNest / MR Notifier
**Date:** 2026-02-23 04:11 UTC  
**Auditor:** Gemini CLI  
**Scope:** Full Repository Analysis  
**Score:** 45/100

---

## 1. Executive Summary
The project is in a fractured state, split across three main directories: `mr_notifier-server` (Hono/TypeScript), `mr_notifier` (Barebones Android), and `forked-version` (Full Android). While the server and `forked-version` show architectural intent (SignalNest), the implementation is plagued by local environment constraints (Termux) and significant logic flaws. The primary Android app (`mr_notifier`) is currently non-functional as an "Event Intelligence System" as it lacks network permissions and server integration.

---

## 2. Critical Issues

### 2.1 logic mixup: Mock vs. Real Mode (Server)
In `mr_notifier-server`, many routes mix in-memory mock logic with database logic.
- **Sync Bug:** `syncRoutes.post('/')` uses `mockSyncState` to calculate the `currentVersion` even in "Real Supabase mode". This will cause sync version desynchronization and data loss in production.
- **Pagination Bug:** `notificationRoutes.get('/')` defines `start` and `end` inside an `if(isMock())` block, but uses them in the Supabase query. This will throw a `ReferenceError` in production.

### 2.2 Missing Internet Permission (Android App)
The `mr_notifier` app lacks the `<uses-permission android:name="android.permission.INTERNET" />` in `AndroidManifest.xml`. It cannot communicate with the server in its current state.

### 2.3 Empty DSL Logic (Server)
The `src/dsl/` directory contains empty subdirectories (`lexer`, `parser`, `compiler`, etc.). The core "Event Intelligence DSL Engine" advertised in the `package.json` description is missing or unimplemented.

---

## 3. Major Issues

### 3.1 Fractured Repository Structure
The project contains two separate Android applications and a server. `mr_notifier` seems to be a "fallback" version created due to Termux limitations, while `forked-version` contains the real logic but is marked as "broken" for local storage in its own `REFACTORING_PLAN.md`.

### 3.2 Lack of Testing
Despite `vitest` being configured in `package.json`, no test files (`.test.ts` or `.spec.ts`) exist in the server source. The Android apps also lack unit/instrumented tests.

### 3.3 No Schema Management
The server's `initDatabase` function is a placeholder that logs success without performing any migrations. There is no automated way to ensure the Supabase/PostgreSQL schema matches the entity definitions in `entities.ts`.

---

## 4. Minor Issues & Technical Debt

### 4.1 Dependency on Developer Environment
The project is heavily influenced by Termux limitations (e.g., avoiding Room/Hilt/KAPT). While understandable for a local hobby project, it prevents the project from reaching "A-grade production standard".
### 4.2 In-Memory Mock Data
The server's mock mode uses non-persistent in-memory maps. Data is lost on every restart, making it unsuitable for even basic staging or integration testing.
### 4.3 Missing CI/CD for Server
There are no GitHub Actions or CI configurations for the server module.

---

## 5. Long-term Risks
- **Scalability:** The current "Sync" logic is naive and relies on simple timestamp/version comparison which may fail under high concurrency or complex edge cases (e.g., clock skew).
- **Maintainability:** The mix of route logic and database calls (lack of Repository/Service layers) will make the server increasingly difficult to test and maintain.
- **Viability:** The missing DSL core means the project's unique selling point (event intelligence) is currently vaporware.

---

## 6. Suggested Roadmap Updates
1. **Unify Android Apps:** Merge the `forked-version` logic into `mr_notifier` and use Build Flavors or DI to handle Termux vs. Production environments.
2. **Fix Real-Mode Logic:** Remove dependencies on `mock-store` from production-mode blocks in the server.
3. **Implement DSL Core:** Prioritize the lexer and parser for the event engine.
4. **Setup Schema Migrations:** Integrate a tool like `prisma` or `supabase-cli` for schema management.
5. **Add INTERNET Permission:** Crucial for app-server communication.
6. **Implement Test Suite:** Start with unit tests for the DSL engine and Auth routes.
