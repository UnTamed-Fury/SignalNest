# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-23 02:02:50 UTC
**Auditor:** Gemini CLI
**Scope:** Full Repository Audit (Server & Mobile)

---

## Executive Summary
The SignalNest project is currently in a state of major architectural transition (Refactoring Phase). The project is moving from an older package (`fury.notifier`) to a modernized Jetpack Compose application (`com.signalnest.app`). The server is also being upgraded to version 3.0.0, introducing Hono and Supabase integration. While the architectural direction is sound (Clean Architecture/MVVM), the implementation is hampered by significant "Work-In-Progress" gaps, most notably the lack of persistent client-side storage and missing core features like the DSL engine.

---

## Critical Issues 🔴

### 1. Client-Side Data Volatility (In-Memory Storage)
The `SignalNestRepository` currently uses `MutableStateFlow` with in-memory lists for all data including Notifications, Notes, and Todos.
- **Impact:** All user data, read statuses, and local annotations are lost when the app process is terminated.
- **Risk:** High. This makes the app unusable for real-world production.
- **Context:** The `REFACTORING_PLAN.md` notes this as a compromise due to "kapt issues on Termux."

### 2. Significant Security Risk: `JWT_SECRET` Exposure
The `ApiClient.kt` file defines a preference key for `JWT_SECRET` and stores it in `DataStore`.
- **Impact:** If this secret is used for symmetric JWT verification on the client, the entire security model is compromised. Secrets should never be stored on the client.
- **Risk:** High.

### 3. Total Absence of Automated Tests
Despite having `vitest` in the server and a test directory structure in the Android app, there are **zero** functional test cases in the entire repository.
- **Impact:** Regressions are guaranteed during refactoring.
- **Risk:** High.

---

## Major Issues 🟡

### 1. Missing Core Feature: DSL Engine
The `mr_notifier-server/src/dsl` subdirectories (compiler, executor, lexer, etc.) are empty.
- **Impact:** The project's stated goal of being an "Event Intelligence System with DSL Engine" is currently unfulfilled at the code level.

### 2. Flawed Sync Logic (State Overwrite)
The `syncNotifications` method in `SignalNestRepository` overwrites local notification objects with server responses if the IDs match.
- **Impact:** Local-only state (like pins or read status if not yet synced) is discarded instead of merged.
- **Risk:** Medium.

### 3. Brittle API Client Lifecycle
The `ApiClient.api` instance is a `lazy` property. If the `baseUrl` is updated via settings *after* the first API call, the `api` singleton will continue using the old URL.
- **Impact:** Users cannot change server URLs without restarting the app.

---

## Minor Issues 🟢

### 1. Notification ID Collision Risk
`SignalNestMessagingService` uses `NOTIFICATION_ID + title.hashCode()` to generate notification IDs.
- **Impact:** While unlikely to cause crashes, title collisions or hash overflows could lead to notifications overwriting each other in the system tray.

### 2. Incomplete Device Registration
The `onNewToken` method in the FCM service is a TODO, meaning the server cannot yet target specific devices for push notifications.

---

## Long-term Risks
- **Termux Constraint:** The decision to optimize for Termux (avoiding `kapt`) is forcing sub-optimal architectural choices (avoiding Room). This will lead to massive technical debt as the project grows.
- **Statelessness vs Persistence:** The struggle between a stateless architecture and the need for local persistence is visible and currently unresolved.

---

## Suggested Roadmap Updates
1. **Immediate:** Implement SQLite persistence in the Android app (using Room with KSP or plain SQLite to bypass Termux `kapt` issues).
2. **Security:** Remove `JWT_SECRET` from client-side storage. Use standard OAuth2 or Session management.
3. **Core:** Implement the DSL lexer and parser to fulfill the project's primary value proposition.
4. **Validation:** Write unit tests for `SignalNestRepository` and server-side `sync` routes.
5. **Lifecycle:** Fix `ApiClient` to properly recreate the Retrofit instance when the `baseUrl` changes.

---

## Score: 45/100
*The project has a clear vision and professional structure, but the lack of persistence and tests, combined with critical security patterns, places it below production standard.*
