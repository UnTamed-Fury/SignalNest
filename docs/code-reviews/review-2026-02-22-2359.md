# Code Audit Report: SignalNest (mr_notifier)
**Date:** 2026-02-22 23:59
**Audit Duration:** 10 minutes
**Auditor:** Gemini CLI

---

## Executive Summary
SignalNest (formerly mr_notifier) is an event intelligence system with a mobile client and a TypeScript-based server. While the project uses modern frameworks (Hono, Supabase, Jetpack Compose), it is currently in an early development phase, characterized by "skeleton" code. Many critical modules (DSL engine, database persistence) are either empty or use volatile in-memory storage. A severe security vulnerability was identified in the data synchronization layer.

---

## Critical Issues (Must Fix Immediately)
1. **Broken Access Control (IDOR):** The `/api/sync` endpoint in the server trusts the `user_id` provided in the request body without verifying it against the JWT token. Any authenticated user can potentially download notifications for any other user by simply changing the `user_id` in the payload.
2. **Zero Test Coverage:** No unit, integration, or E2E tests were found in the entire repository. This makes the project extremely fragile and impossible to refactor safely.
3. **Skeleton Modules:** The core value proposition—the DSL engine (`src/dsl/`)—consists of empty directories. The server is currently just a basic CRUD API with a health check.

---

## Major Issues
1. **Volatile Storage (Android):** The `SignalNestRepository` in the `forked-version` app uses `MutableStateFlow` as its primary storage. All notifications, notes, and todos are lost when the app process is killed. The promised Room integration is missing.
2. **Hardcoded Logic:** Both the Android app and the server routes contain hardcoded user IDs (often `0`), indicating that multi-tenant logic is not yet fully implemented.
3. **Mock Mode Risk:** The server automatically falls back to a "Mock Mode" if Supabase credentials are missing. While useful for dev, a misconfiguration in production could lead to data being silently stored in-memory and lost on restart.

---

## Minor Issues
1. **Missing Dev Tools:** `package.json` references `eslint` and `vitest` in scripts, but the dependencies or configurations (e.g., `.eslintrc`) are missing or incomplete.
2. **Inconsistent Naming:** The project is transitioning from `mr_notifier` to `SignalNest`, leading to inconsistent naming across directories and documentation.
3. **Documentation Gaps:** Code comments often refer to features "coming soon" or "placeholder," which contradicts some of the high-level README claims.

---

## Long-term Risks
- **Scalability:** The server's mock store uses global `Map` objects. Under high load or with many users, this will lead to memory exhaustion and performance degradation.
- **Maintainability:** Without a test suite, as the DSL complexity grows, the risk of regressions in the event-processing logic is 100%.

---

## Suggested Roadmap Updates
1. **Security Patch:** Implement JWT-based user verification in all `/api/` routes.
2. **Persistence (Android):** Replace in-memory storage with Room database for notifications, notes, and todos.
3. **DSL Implementation:** Prioritize the lexer and parser for the event intelligence DSL.
4. **Testing Infrastructure:** Initialize Vitest for the server and JUnit/Compose-Test for the Android app. Target 50% coverage for core logic.
5. **CI/CD Alignment:** Ensure GitHub Actions are configured to run linting and tests (once added) on every PR.

---

## Project Score: 35/100
**Grade: F (Not Production Ready)**
*The project has a clear vision and good tech choices, but the combination of critical security flaws, lack of persistence, and zero tests makes it unsuitable for any environment beyond local prototyping.*
